## 二、逐层详解

### 层1: DeepSeek API 调用

| 维度 | 变更前 | 变更后 | 影响 |
|------|--------|--------|------|
| 单次Demo生成调用次数 | 1次 | 1-4次(含重试) | 平均1.3次 |
| 每次max_tokens | 16384 | 32768 | 响应更长但更完整 |
| 单次耗时 | ~27s | 27-90s(含重试) | 用户等待↑ |
| Token消耗 | ~8K/次 | ~12K/次(含重试) | 成本↑50% |

**缓解**: 第一次就成功的场景占80%，平均耗时增加有限。

### 层2: Demo 生成流程

**变更前**:
```
用户点「生成预览」→ DeepSeek → 返回HTML → 存DB → 展示
                                  ↓ 失败
                              报错「生成失败」
```

**变更后**:
```
用户点「生成预览」→ DeepSeek → 闸门1(结构) → 闸门2(属性) → 闸门3(隔离) → 闸门4(验收) → 存DB → 展示
                     ↓ 不过    ↓ 不过       ↓ 不过       ↓ 不过
                   重试(附原因) 自动补属性   快照回退    强化Prompt重试
                     ↓ 3次仍失败
                   标记 NEEDS_HUMAN
```

### 层3: HTML 合并/提取

| 函数 | 变更 |
|------|------|
| `extractHtml()` | +清洗markdown标记、+完整性校验 |
| `mergeModifiedHtml()` | +非目标模块回退、+script块保护 |
| `extractModules()` | +新元素自动注入data-module-key |
| `compressHtml()` | +保留data属性不被替换 |

### 层4: 质量门禁

新增4项检查：

| 检查项 | 判定标准 |
|--------|----------|
| HTML标签闭环 | `<!DOCTYPE` 开头 + `</html>` 结尾 |
| 无Markdown残留 | 不含 \`\`\` 标记 |
| data-module-key完整 | 所有模块元素含该属性 |
| 非目标模块一致 | 对比快照，其他模块内容无变化 |

### 层5: 版本快照

| 变更 | 说明 |
|------|------|
| 快照时机 | 每次修改**前**自动创建（当前已有时机不变） |
| 回退用途 | 闸门3失败→从快照恢复非目标模块 |
| 存储 | 不变，已有 demo_snapshots 表 |

### 层6: 错误模式库

新增3个模式：

| 模式key | 触发条件 |
|---------|----------|
| `html_truncated` | 响应不含 `</html>` 或长度 < 500 |
| `attribute_lost` | data-module-key 数量减少 |
| `module_polluted` | 非目标模块内容被改动 |

### 层7: 前端体验

| 页面 | 当前 | 变更后 |
|------|------|--------|
| Demo页 | 显示「AI正在生成预览...」 | +「正在质检...」「自动修复中(1/3)...」 |
| 评估页 | 一次失败即停止 | 自动重试，进度推送重试次数 |

### 层8: 自迭代

**正面影响**: autoFix 复用闸门，减少「修了A坏了B」的无效迭代。

**负面影响**: 每轮迭代耗时增加（因为多了质检步骤），但总轮数会减少。

## 三、不改动的部分

| 模块 | 原因 |
|------|------|
| 规格确认(Specification) | 不涉及AI调用 |
| 决策引擎(DecisionEngine) | 纯规则判断 |
| 系统提醒(WarningService) | 只读匹配 |
| 测试部署(TestDeployment) | Docker操作 |
| CC Bridge | 降级路径不受影响 |
| Prisma Schema | 无新模型 |

## 四、风险矩阵

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| 重试导致用户等待过久 | 中 | 体验下降 | 前端展示进度，超2次提示用户可选跳过 |
| Token成本显著增加 | 低 | 运营成本↑ | 80%场景一次通过，平均增加有限 |
| 闸门规则过严导致误杀 | 低 | 正常生成被拒绝 | 闸门阈值可配置，初期宽松逐步收紧 |
| 快照回退丢失用户修改 | 极低 | 数据丢失 | 只在AI修改后回退AI污染部分，不动用户手动修改 |

---

## 五、总结

| 维度 | 影响程度 | 说明 |
|------|----------|------|
| 架构 | ⬜ 无影响 | 全部在现有服务层增强 |
| 数据库 | ⬜ 无影响 | 不新增表 |
| API端点 | ⬜ 无影响 | 不新增端点 |
| 前端 | 🟡 小幅 | 进度展示文案调整 |
| AI调用 | 🔴 显著 | 每次调用多1-3次重试，token消耗↑ |
| 生成耗时 | 🟡 中等 | 平均增加10-30秒 |
| 生成质量 | 🟢 显著提升 | 15类问题自动修复 |
| 运维成本 | 🟡 小幅 | Token费用增加，但减少人工修复时间 |
