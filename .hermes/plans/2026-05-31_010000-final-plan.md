### 1.3 改造 runAutoIterate() 核心循环

**文件**: `modules/delivery/delivery.service.ts` (行 631-710)

**旧逻辑**: `analyzeSilent()` → `qualityGate` → `computeMixedScore(AI*40%+Q*30%+F*30%)`

**新逻辑**:

```
每轮迭代:
  1. SENSE — SensorService.runAll(projectId)
     → FusedReport { overallScore, l1/l2/l3分项, recommendations,
                     stopIteration, missingRequirements, hallucinations }
  
  2. DECIDE — 退出检查:
     - stopIteration=true 且 score ≥ 90 → 达标，弹确认进入交付
     - stopIteration=true 且 score < 90 → 用户介入
     - 连续3轮 score 不变 → 用户介入
  
  3. FIX — 提取可修复问题:
     - 缺失需求 → 构造修复 prompt → Cloudecode 定向修复
     - 疑似幻觉 → 重新生成相关模块
     - L1 静态失败 → Cloudecode 修复 HTML
     - 修复后 → 保存 DemoSnapshot → 进入下一轮
```

**注入依赖**: `SensorService` 加入 DeliveryService 构造函数。

### 1.4 SSE 事件格式升级

**前端文件**: `projects/[id]/evaluation/page.tsx`

```typescript
// 新 round_result 结构
{

  checks: { passed, total }

```

前端进度条改为三段式：L1蓝/L2绿/L3紫，下方显示"需求覆盖率 75%（9/12 验收标准通过）"。

## Phase 2: 企业级能力补齐（2-3天）

### 2.1 RBAC — PlanGuard

**新增文件**:
- `common/guards/plan.guard.ts` (30行)
- `common/decorators/required-plan.decorator.ts` (10行)

基于现有 `User.plan` 字段（free/pro/enterprise）做功能门控：

| plan | 允许 |
|------|------|
| free | 创建项目、需求描述、查看 Demo |
| pro | + 自迭代、导出源码 |
| enterprise | + 全栈交付、CI/CD webhook |

**改动文件**: `delivery.controller.ts`、`deploy.controller.ts` 端点加 `@RequiredPlan('pro')`

### 2.2 用户操作审计

**Schema**: `DecisionLog` 加 `userId String?` 字段

**新增文件**: `common/interceptors/audit.interceptor.ts` (40行)  
全局拦截 POST/PUT/DELETE，自动写入 `stage='user_action'` 的决策日志。

**改动文件**: `app.module.ts` 注册全局拦截器

### 2.3 CI/CD Webhook

**新增文件**:
- `modules/webhook/webhook.controller.ts` (60行)
- `modules/webhook/webhook.module.ts` (10行)

接收 GitHub Webhook push 事件 → 验证签名 → 触发 PipelineService 自动交付。

**改动文件**: `app.module.ts` 注册 WebhookModule

### 2.4 速率限制

安装 `@nestjs/throttler`，`app.module.ts` 中注册全局限流（60秒/30次）。企业版可提限。

### 2.5 监控告警

利用 Hermes cronjob + 现有 `SensorService`，每 30 分钟自动执行平台健康检查，score < 60 时通知。

**改动**: 一个 cronjob 配置，无需改代码。
