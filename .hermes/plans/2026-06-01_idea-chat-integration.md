# 需求访谈：对话窗口 vs 独立页面

**日期**: 2026-06-01
**状态**: 方案

---

## 对比

| 维度 | 对话窗口 (推荐) | 独立页面 (当前) |
|------|----------------|-----------------|
| 用户习惯 | ✅ 已有项目聊天页，自然延伸 | ⚠️ 需要学习新界面 |
| 上下文 | ✅ 回答直接存为聊天记录 | ⚠️ 回答存在 structuredRequirement 不可见 |
| 进度感 | ⚠️ 需额外设计进度提示 | ✅ 进度条、阶段图标清晰 |
| 实现复杂度 | 中：修改现有 chat 页 | 低：已基本完成 |
| 与平台一致性 | ✅ 同 Discovery 引擎一致 | 独立页面型 |

---

## 推荐：对话窗口

原因：
1. 平台已有的项目详情页(`/projects/[id]`)就是对话窗口——用户说想法，AI回应
2. 需求访谈本质就是"AI引导用户说清楚需求"，天然适合对话形式
3. 不用切换页面，降低用户认知负担
4. 访谈记录和项目消息合为一体，不分散

## 方案

```
用户进入项目 → 看到对话窗口
  → 如果还没做过需求访谈，AI主动说：
    "你好！我来帮你把这个想法理清楚。我们一步一步来——"
  → 逐题发问，用户回答
  → 每次回答后 AI 追问下一题
  → 完成全部阶段后：
    "需求访谈完成！我已经生成了结构化需求文档，你可以查看方案了。"
    → 自动跳转或提示进入方案页
```

## 改动

| 文件 | 改动 |
|------|------|
| `project/[id]/page.tsx` | 页面加载时调 `/idea`，未开始则自动发第一条问题 |
| `project/[id]/page.tsx` | 用户发送消息时，如果处于访谈中则作为回答提交 |
| `idea-interview.service.ts` | 加 `startFromChat()` 方法，适配合并进chat消息格式 |
| ~~`project/[id]/idea/page.tsx`~~ | 可删除或保留作为备选入口 |

## 工作量

~30 分钟。改动集中在 1 个页面 + 1 个服务。
