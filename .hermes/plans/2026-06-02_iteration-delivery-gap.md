# 自迭代后交付断层 — 根因分析与解决方案

> 2026-06-02 | 4 个问题的根因和解决

---

| 指标 | 迭代前 | 迭代后 |
|------|--------|--------|
| Demo HTML | 18,030 chars | **33,640 chars** (+86%) |

| 代码生成 | ❌ 未触发 | ❌ 未触发 |

## 二、四个问题的根因

### Q1: 为什么没有代码存在？

**根因**: 自迭代 ≠ 代码生成。自迭代只改进 Demo HTML，代码生成是独立步骤。

```
自迭代:  传感器分析 → autoFix → 保存到 demoHtml → 循环
代码生成: 用户点击"开始交付" → 分步生成 → 保存文件 → 部署
```

自迭代完成后，用户没有收到"现在可以交付了"的引导，也没有自动触发代码生成。

### Q2: 每次自迭代的改进是否存下来了？

**是的，每轮都存了。**

```typescript
// delivery-iteration.service.ts:256-259
await this.prisma.project.update({
    where: { id: projectId },
    data: { demoHtml: newDemo },  // ← 每轮都更新
});
```

证据：Demo HTML 从 18KB 增长到 33KB，每轮 autoFix 都在生成更大的 HTML。

### Q3: 为什么还有 11 条建议就停止迭代了？

**根因**: `MAX_ROUNDS = 10` 上限。迭代不是因为"没有建议了"停止，而是因为"达到最大轮数"。

```typescript
// delivery-iteration.service.ts:134
const MAX_ROUNDS = 10;
```

建议数量 ≠ 应该继续迭代的信号。传感器在每轮都能找到新问题（HTML 变了，传感器发现的新问题也不同），建议不会归零。

### Q4: 自迭代完是不是没有代码生成，所以 Demo 没改进？

**Demo 改进了（+86%），但代码没生成。** 这是**流程断层**：

```
用户期望: 自迭代 → 代码自动生成
实际:     自迭代 → Demo 改进 → (用户需要手动去交付页点按钮)
```

---

## 三、解决方案

### 方案 A: 自迭代完成后自动触发交付（推荐）

```
自迭代完成 (done/stuck)
    ↓
显示弹窗: "Demo 已优化至 XX 分，是否开始终稿交付？"
    ├─ [开始交付] → 自动跳转交付页 + 触发代码生成
    └─ [稍后再说] → 留在评估页
```

**改动**: `evaluation/page.tsx` — 在 `done`/`stuck` 事件处理中增加"开始终稿交付"按钮，直接调用 delivery API。

### 方案 B: 降低 MAX_ROUNDS 感知，增加质量判定

```
当前: MAX_ROUNDS = 10, 到 10 轮就停
改为: MAX_ROUNDS = 10, 但提前停止条件改为 "连续3轮评分无改善 且 当前评分 >= 70"
```

**改动**: `delivery-iteration.service.ts` — 修改停止条件。

### 方案 C: 交付页增加"自迭代后一键交付"快捷入口

```
交付页 → 检测到有 demoHtml 且 status=demo_ready 或 completed
    → 自动显示"基于当前 Demo 交付"按钮
    → 不需要用户先跑自迭代再跑交付
```

## 四、建议执行

**立即执行方案 A**（改动最小，体验改善最大）：自迭代完成弹窗增加"开始终稿交付"按钮。

**然后执行方案 B**：降低无意义迭代轮次。
