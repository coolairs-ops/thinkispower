# Think-is-power 需求设计 → 全栈交付 全链路改造计划

> **For Hermes:** Use subagent-driven-development + @quich-engineering 协同实施。
> 本计划涵盖前端评估页改造、设计建议系统、自迭代引擎、生成树可视化和全栈交付终稿。

**目标：** 用户只需给出功能/界面/权限/流程要求，平台自动完成技术选型、代码生成、部署，中间通过设计建议+自迭代达到100%完整度后终稿交付。

## 一、需求澄清改造（只问用户关心的）

### 改造范围
`message.service.ts` 和 `clarify.service.ts`

### 具体改动
Hermes 的澄清 prompt 加入过滤规则：
- **保留给用户的问题类别：** 功能范围、页面/导航布局、字段定义、操作流程、权限角色
- **平台自决类别：** 接口协议(GraphQL vs REST)、数据库选型、部署方案、框架版本、中间件选型、缓存策略、CI/CD配置

### 实现
在 `CLARIFY_PROMPT` 中加入：
```
你是产品需求分析师。只能询问以下4类问题：
1. 功能范围 — 需要哪些核心功能模块
2. 页面/导航 — 几级菜单、页面布局偏好
3. 字段定义 — 核心数据字段和关系
4. 操作流程 — 创建、审核、发布等业务流程
5. 权限角色 — 用户角色和权限划分
禁止询问技术实现细节（数据库、框架、接口协议等），这些由平台自动决定。
```

## 三、自迭代评估引擎（核心改造）

### 流程
```
方案确认 → 自动评估 → 得分 < 100% → 自动修复迭代
  ↓ 评分不动三轮                                     ↓ 每轮在涨
  用户介入（给选项）                                   继续迭代
  ↓                                                  ↓
  用户决策后继续                                      直到 100%
  ↓
  100% → 通知用户「方案已达100%」→ 用户确认
  ↓
  拒绝 → 用户给方向 → 继续迭代
  ↓ 确认
  终稿全栈交付（带生成树动画）
```

### 修改文件
- `apps/api/src/modules/delivery/delivery.service.ts` — 新增 `autoIterate()` 方法
- `apps/api/src/modules/delivery/delivery.controller.ts` — 新增 `POST auto-iterate/start` `POST auto-iterate/decide`
- `apps/web/src/app/projects/[id]/evaluation/page.tsx` — 改造为自迭代面板

### `autoIterate()` 逻辑
```
iterate(projectId, maxRounds=20):
  lastScore = 0
  stuckCount = 0
  for round in 1..maxRounds:
    // 1. 执行评估
    analysis = hermes.analyzeSilent()
    quality = qualityGate.runAllChecks()
    score = quality.score
    
    // 2. 检查是否100%
    if score >= 100: return { done: true, score: 100 }
    
    // 3. 检查是否停滞
    if score <= lastScore:
      stuckCount++
      if stuckCount >= 3: return { stuck: true, score, stuckCount, analysis }
    else:
      stuckCount = 0
    
    lastScore = score
    
    // 4. 自动修复：选最高优先级风险，调 Cloudecode 修复
    topRisk = analysis.risks.sort(by severity)[0]
    cloudecode.executeTaskForProject(projectId, topRisk.fixContent)
    
    // 5. 等待修复完成
    sleep(15s)
  
  return { stuck: true, score, maxRoundsReached: true }
```

### 用户介入界面
当 `stuck=true` 时，前端显示：
- 当前完整度分数 + 三维度详情
- 「先看看 Demo 预览」按钮 → 打开 demo 页
- 两套方案选项（方案A / 方案B），用户选一个
- 或用户自由输入改进方向

### SSE 进度流
自迭代过程中，前端通过 SSE 实时接收每轮结果：
```
event: iteration
data: {"round": 3, "score": 78, "delta": +5, "topFix": "添加表单验证逻辑"}

event: stuck
data: {"round": 6, "score": 88, "stuckCount": 3}
```

## 四、生成树可视化（全新组件）

### 位置
交付页 `/projects/[id]/delivery` 上方，占 60% 宽度的大窗

### 双模式切换
窗顶两个按钮：
- **「文件结构树」** — 按目录层级展示生成的文件
- **「阶段流程树」** — 按时间阶段展示进度

### 文件结构树
```
┌─ src/
│  ├─ pages/        [蓝色枝]
│  │  ├─ login.tsx
│  │  └─ dashboard.tsx
│  ├─ api/          [绿色枝]
│  │  └─ user.ts
│  └─ components/   [橙色枝]
├─ Dockerfile       [紫色枝]
├─ docker-compose.yml
└─ nginx.conf
```

### 阶段流程树
```
需求分析 ──→ 前端生成 ──→ 后端生成 ──→ 数据库 ──→ 部署 ──→ 打包
 [蓝色]      [绿色]      [橙色]      [红色]    [紫色]   [金色]
```

### 核心规则
- 从左往右生长（根在左，叶在右）
- 不同枝干不同颜色（最少5色）
- 正在生成的节点闪烁动画
- 已完成的节点 ✓ 标记
- 失败的节点 ✗ 红色

### 新增文件
- `apps/web/src/components/generation-tree.tsx` — 通用生成树组件
- `apps/web/src/components/file-tree-view.tsx` — 文件结构树渲染器
- `apps/web/src/components/phase-tree-view.tsx` — 阶段流程树渲染器

### 修改文件
- `apps/web/src/app/projects/[id]/delivery/page.tsx` — 嵌入生成树

## 五、终稿全栈交付（100%路径）

### 与 Demo 的区别
- Demo：单 HTML 文件预览（快，27s）
- 终稿：全栈代码 + Docker + Nginx + 部署 + ZIP下载

### 交付物
```
delivery/
├── frontend/          # Next.js 14 + TypeScript
│   ├── src/pages/
│   ├── src/components/
│   └── package.json
├── backend/           # NestJS + Prisma + TypeScript
│   ├── src/modules/
│   ├── prisma/schema.prisma
│   └── package.json
├── docker-compose.yml
├── nginx.conf
├── Dockerfile
├── README.md
└── .gitignore
```

### 部署预留
在 `docker-compose.yml` 中预留：
```yaml
services:
  app:
    build: .
    ports: ["APP_PORT:3000"]
    environment:
      - DATABASE_URL=${DATABASE_URL}
      - DEPLOY_KEY=${DEPLOY_KEY}
  db:
    image: postgres:16
    volumes: ["pgdata:/var/lib/postgresql/data"]
volumes:
  pgdata:
```

部署接口预留：`POST /api/deploy/:projectId/launch` — 接收部署目标（服务器IP/容器/VM），返回访问 URL。

### 修改文件
- `apps/api/src/services/delivery-orchestrator.service.ts` — 全栈代码生成 + 部署
- `apps/api/src/modules/deployment/deployment.service.ts` — 新增 launch 端点
