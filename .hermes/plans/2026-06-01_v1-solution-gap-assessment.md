# V1.0 解决方案 vs 现状差距评估

**日期**: 2026-06-01  
**目的**: 评估《思想动力》V1.0 详细解决方案与当前平台的差距，量化调整幅度、影响和工作量。

---

## 一、现状 vs 目标对照表

V1.0 方案定义了 14 个核心模块。以下逐一对照现有实现：

| # | 模块 | 目标描述 | 现有状态 | 差距 | 方向调整 |
|---|------|----------|----------|------|----------|
| 1 | 想法输入 | 行业识别、模板推荐、清晰度判断 | ⚠️ 项目创建页有基础输入框，无行业识别/模板推荐 | 中 | 新增 |
| 2 | 需求澄清 | 带示例的问题、快捷选项、自由输入 | ✅ Discovery Engine + CompletenessChecker (7维度) | 小 | 增强 |

## 二、差距分级

### 🔴 重大差距（结构性缺失，需从零建设）

**模块 6：规格确认** — 影响最大
- 原因：当前整个流程缺少"确认后锁定"的契约节点。没有规格确认，自动开发就没有边界——用户会持续改需求，平台永远无法判定"是否按规格交付"。
- 工作量估算：后端 8-10 人日 + 前端 5-6 人日 = **~15人日**
- 需要新建：Specification 模型、冻结/版本号机制、确认页、验收场景定义
- 影响面：涉及 project 状态机扩展、delivery 评估逻辑、feedback 判定（bug vs变更）

**模块 5：下一步建议（决策树）** — 护城河核心
- 原因：方案中定义的"根据状态判断继续/暂停/缩小范围"逻辑完全缺失，这是工程控制论闭环的关键节点。
- 工作量估算：后端 5-7 人日 + 前端 3-4 人日 = **~10人日**
- 需要新建：DecisionEngine 服务、状态→建议映射规则、前端展示组件
- 影响面：依赖 completeness-checker、error-pattern、estimate 数据

### 🟡 中等差距（有基础但需大幅增强）

**模块 4：系统提醒（错误模式库前端化）** — 5人日
- ErrorPattern 模型已存在但未生成用户可读的提醒文案
- 需要在需求澄清/预测阶段主动弹出提醒

**模块 9：测试环境自动部署** — 10-12人日
- 需要：Docker 动态构建、临时域名分配、HTTPS、健康检查流水线
- 这涉及基础设施改动，可能需要 Traefik/Caddy 反向代理

**模块 8：自动开发执行流程化** — 5-7人日
- Cloudecode 有能力生成全栈代码，但需要从"企业交付手动触发"变为"规格确认后自动触发"

**模块 12：技术门禁扩展** — 8-10人日
- 从4项质量检查扩展到15项+（API契约、安全扫描、移动端适配、Docker构建等）

### 🟢 小幅差距（增强现有能力）

**模块 1** — 2人日：输入框增强行业识别/模板推荐
**模块 2** — 2人日：增强选项示例的UX
**模块 3** — 3人日：人民币成本映射
**模块 10** — 4人日：测试环境内嵌反馈浮窗
**模块 11** — 3人日：3次上限+人工升级
**模块 13** — 2人日：完善文档/备份说明
**模块 14** — 3人日：预测校准对比

## 三、工作量总估算

| 级别 | 模块数 | 总人日 |
|------|--------|--------|
| 🔴 重大 | 2 (规格确认 + 决策树) | ~25 |
| 🟡 中等 | 4 (错误模式/测试部署/自动开发/技术门禁) | ~32 |
| 🟢 小幅 | 6 | ~16 |
| **合计** | **12 个有差距的模块** | **~73 人日** |

以 1 人全职计算约 **3.5 个月**。考虑到前后端并行开发，实际可压缩到 **2-2.5 个月**。

## 四、影响分析

### 架构影响

1. **状态机重构** — 方案定义了 17 种项目状态（IDEA_CREATED → REVIEWED），当前只有约 8 种。需扩展 `status-mapper.service.ts` 的 TRANSITIONS。
2. **新增 Prisma 模型** — Specification, DecisionLog, ErrorPatternMatch, TestDeployment 至少 4 个新表。
3. **新增 API 模块** — specification, decision-tree, test-deployment 至少 3 个新模块。
4. **前端新增页面** — spec-confirm, test-env-deploy, project-review 至少 3 个新页面。

### 不影响的现有能力

- 传感器系统（L1/L2/L3）— 不受影响，继续作为质量评估基础
- Demo 生成 + Cloudecode 全栈交付 — 不受影响
- Discovery 对话引擎 — 小幅增强，不重构
- 交付评估 + 修复队列 — 小幅增强（规格确认会改变判定逻辑）

### 最大风险

- **规格确认是分水岭**：引入后，feedback 的判定逻辑需要从"用户说改就改"变为"判断是 bug 还是变更请求"。这会影响 acceptRiskFix + autoFix 的行为。
- **测试环境部署**依赖基础设施：需要域名管理、HTTPS 证书、容器编排升级。

## 五、交付完整度评估

基于当前平台状态和 V1.0 方案差距，分阶段完整度预估：

| 阶段 | 覆盖模块 | 完整度 | 说明 |
|------|----------|--------|------|
| 当前 | 1(部分)2(✅)3(部分)7(✅)8(部分)10(部分)11(部分)12(部分)13(✅)14(部分) | **~45%** | 核心链路通但不完整 |
| +规格确认(模块6) | +6 | **~55%** | 补齐最大结构性缺口 |
| +决策树(模块5) | +5 | **~65%** | 工程控制论闭环成型 |
| +测试部署(模块9) | +9 | **~75%** | 用户可在线验收 |
| +错误模式前端(模块4)+技术门禁扩展(模块12) | +4,12 | **~85%** | 质量和风险体系完整 |
| +自动修复边界(模块11)+复盘校准(模块14) | +11,14 | **~92%** | 接近方案定义目标 |
| +其余小幅模块 | 1,2,3,10,13 | **~98%** | 方案定义范围内完整 |

**结论**：当前 ~45%，做完全部 12 个缺口可达 ~98%。但 "100% 完整度" 需要实际运营数据反哺（估算校准、行业模板积累），这不是纯开发能达成的。

## 六、建议优先级

V1.0 方案自己建议的下一步顺序是对的：

```
第 9 步（模块4：系统提醒）→ 第 10 步（模块5：下一步建议）→ 第 14 步（模块6：规格确认）
```

但我建议微调顺序，把规格确认提前：

```
1. 模块 6：规格确认（最大结构性缺口，15人日）
2. 模块 5：决策树（护城河，10人日）
3. 模块 4：错误模式前端化（5人日）
4. 模块 9：测试环境部署（12人日，基础设施依赖）
5. 模块 12：技术门禁扩展（10人日）
6. 其余小幅模块（~16人日）
```

原因：没有规格确认，决策树的"进入开发"判断没有依据。先有规格冻结，再有决策逻辑，最后是错误提醒优化。

---

## 七、一句话结论

**调整幅度**：12/14 模块有差距，其中 2 个重大（规格确认、决策树），4 个中等，共需 **~73 人日**。

**影响**：不影响现有核心链路，但规格确认会改变 feedback 判定逻辑。架构上需新增 4+ Prisma 模型、3+ API 模块、3+ 前端页面。

**完整度**：当前 ~45%，开发完成后可达 ~98%（方案定义范围内）。真正的 100% 需要运营数据积累。

**建议**：规格确认 → 决策树 → 错误模式 → 测试部署，分 4 个阶段推进。
