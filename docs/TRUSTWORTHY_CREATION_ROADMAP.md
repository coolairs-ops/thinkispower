# 可信创造商业化改造路线图

> 目标：把现有 AI 生成与交付平台，改造成“通用智能产品工厂 + 行业根据地 + 持续守护服务”的商业产品。
> 核心戒律：持续照看、双模式、垂直根据地。
>
> **目标架构（底座/规则包/三层生成/适配器/编排，含现状/未来态/触发条件）见 [ADR-0002 平台目标架构](architecture/ADR-0002-platform-target-architecture.md)。** 本路线图给阶段节奏，ADR-0002 给目标态与建设触发条件，互为里子。

## 1. 产品定位

思想动力不再只定位为“一次性生成软件”的工具，而是：

**为企业快速创建、可信交付、持续守护内部数字系统的 AI 产品工厂。**

商业承诺分为三段：

1. **快速创造**：从想法、文档、表格、截图、原型中生成可确认的业务规格与 Demo。
2. **可信交付**：每个功能都能追溯来源、验证通过率、测试证据和变更记录。
3. **持续守护**：上线后持续监控、备份、诊断、修复和迭代，按产品实例长期收费。

## 2. 总体架构

### 2.1 四个产品面

| 产品面 | 面向对象 | 核心价值 | 现有基础 | 改造重点 |
|---|---|---|---|---|
| Create 创造模式 | 业务人员 / 产品人员 | 用业务语言创建系统 | 项目创建、需求澄清、资料导入、Demo | 重做低噪音叙事界面、透明度分级 |
| Studio 监护模式 | 技术人员 / 创客 / 交付人员 | 可介入、可接管、可审计 | Task、Delivery、Sensors、Snapshots | 任务看板、决策日志、代码/测试预览 |
| Guardian 守护中心 | 企业管理员 / 运营 | 持续照看、稳定运行 | 部署、验收报告、部分质量检查 | 监控、备份、巡检、自动修复、健康报告 |
| Industry Kit 行业包 | 解决方案团队 | 行业复制和护城河 | Specification、导入、验收场景 | 行业术语、流程、模块、连接器、验收库 |

### 2.2 通用内核

现有代码中应保留并收口为“可信创造内核”的资产：

- Project 生命周期：创建、需求、方案、Demo、反馈、交付。
- Professional Import：资料上传、解析、需求理解、规格物化。
- Specification：业务规格、验收场景、版本和变更记录。
- Acceptance Report：通过率、证据、人工确认、交付门控。
- Sensors：L1 静态、L2 运行、L3 语义、交叉验证。
- Demo Snapshot：版本快照和回滚基础。
- LLM Gateway：私有化与数据不出域的统一 AI 出口。
- QueueModule / BullMQ：长任务持久化地基。
- Delivery Control：SecurityGate、ExecutorRouter、双消息结构。

## 3. 分阶段执行路线

### Phase 0：可信创造内核收口（当前优先）

目标：让现有底座稳定、可测、可继续改造。

交付物：

- 后端测试全绿，前端 lint 可非交互运行。
- 清理敏感信息、临时文件、补丁脚本和构建缓存。
- 所有 AI 调用逐步收口到 LLM Gateway。
- Demo、导入、交付长任务进入 BullMQ。
- 公开下载、部署预览、webhook 加安全边界。
- 多租户访问由 userId 过渡到 org scope。

验收标准：

- `npm run build` 前后端通过。
- `npm test -- --runInBand` 后端通过。
- 新增/变更的核心行为有单测。
- 不再出现明文密钥、临时 token、入库缓存文件。

### Phase 1：双模式产品外壳

目标：同一套底层过程，投影为两套界面语言。

交付物：

- Create 创造模式：对话式故事板、资料导入、Demo 批注、业务验收。
- Studio 监护模式：任务 DAG、执行日志、决策记录、代码/测试预览、人工接管。
- 透明度协议：L1 业务摘要、L2 一阶揭示、L3 技术细节。
- 业务语言到技术语言映射：每个功能有 businessName、technicalPlan、verificationEvidence。

验收标准：

- 非技术用户全流程不暴露内部技术词。
- 技术用户可看到任务、变更、验证、回滚点。
- 同一条业务指令在两个模式下展示一致但粒度不同。

### Phase 2：持续守护服务

目标：把“一次性交付”升级为“产品实例长期照看”。

交付物：

- Guardian 守护中心。
- 产品实例健康分：可用性、错误、备份、验收回归、依赖风险。
- 自动巡检任务：定时运行关键验收场景。
- 分级修复：提醒、建议修复、确认修复、低风险自动修复。
- 月度健康报告。
- 守护套餐：基础、专业、企业。

验收标准：

- 每个已交付项目都有守护状态。
- 守护任务可追溯、可暂停、可回放。
- 修复动作有影响范围、验证结果和回滚点。

### Phase 3：行业根据地

目标：通用底座不泛卖，先压到一个行业形成可成交方案。

行业包标准结构：

- 行业术语表。
- 领域实体模型。
- 标准流程模板。
- 业务能力组件。
- 页面与报表模板。
- 验收场景库。
- 合规规则。
- 连接器。
- 示例数据。
- 常见失败模式与修复策略。

首个行业选择原则：

- 高频流程、强定制、资料多、付费意愿明确、可复用模块多、售后维护痛点强。
- 候选：连锁门店、制造巡检、园区物业、外包交付、医疗/药企合规流程。

验收标准：

- 至少 10 个行业模块。
- 至少 30 个流程模板。
- 至少 100 条行业验收场景。
- 至少 3 个真实试点项目沉淀可复用资产。

### Phase 4：商业化与规模化运营

目标：从项目制交付走向可运营平台。

交付物：

- 租户套餐、配额、限流、用量计量。
- AI token / GPU 时间 / 存储 / 守护任务计量。
- 支付或合同订阅模型。
- 私有化部署包和数据流向审计报告。
- 可观测性：结构化日志、任务追踪、告警。

验收标准：

- 每个租户可独立计量、限额、审计。
- SaaS 与私有化部署共用同一内核。
- 客户可按“项目搭建费 + 守护订阅 + 行业包增值”付费。

- `AI_MODE=local` 下增加端到端外呼阻断测试。

### 0.4 长任务持久化

- Demo 和导入已进入 BullMQ，继续迁移交付评估、自迭代、部署任务。
- 将进程内 SSE/事件逐步替换为 Redis pub/sub 或持久化事件流。

## 5. 决策原则

1. **不推倒重来**：保留现有闭环资产，围绕商业路线收口。
2. **先稳定再扩张**：测试、安全、队列、多租户优先于新增炫技功能。
3. **双模式共底座**：界面可以两套，任务、规格、验收、日志必须一套。
4. **行业包先于平台梦**：通用能力先服务第一个根据地行业。
5. **持续守护可收费**：每个上线产品都必须进入 Guardian 生命周期。
