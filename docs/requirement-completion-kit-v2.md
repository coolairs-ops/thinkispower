# 需求自动补全工具包 v2（在 v1 基础上的专业升级层）

> 关系：v1（`requirement-completion-kit.md`，原作）是地基——IR 脊柱、阶段1 规则派生、清单分类法、原型库、若依能力锚定、多角色评审、卡片人采纳——**全部采纳，不推翻**。
> v2 只加四件，堵 v1 的结构性盲点，把"补得多"升级为"补得准、补得全、补得可信、补得可续"：
> - **A** IR 完备性批判器 + 锚定"30题的补集"——堵 v1 最大盲点（只能在 IR 内部找缺口）
> - **C** 按"来源"定信任，弃用 LLM 编造的 confidence
> - **D** 区分「自动补默认 / 回去问用户 / 仅提示」——补得准
> - **E** 两遍补全（生成前 + 生成后），与 ADR-0005 建造回路合流——补得可续
>
> 对齐 [ADR-0002] 原则①（固定归确定性、LLM 只补）与 [ADR-0005] 自治建造回路。后台若依集成不在本工具包范围（v1 第7节同此约定）。

---

## 0. 升级后流水线

```
30题采集结果
   │
   ▼
[阶段0] 建模 → IR（v1，不变）
   │
   ▼
[阶段0.5｜新A] IR 完备性批判 ──► 整块缺失(实体/页面/流程/角色/维度) + 30题补集扫描
   │         （堵"IR 没捕获就扫不出"的天花板；召回的真杠杆）
   ▼
[阶段1] 规则派生（v1，不变，100%召回的硬必备）
   │
   ▼
[阶段2] 清单驱动扫描（v1）  ┐
[阶段3] 多角色评审（v1）    ┤─► 每条 gap 打【来源标签｜新C】
[阶段4] 一致性+若依可实现性（v1）┘
   │
   ▼
[阶段4.5｜新D] 处置分类：autofill / ask / info
   │         （确定的自动补；要决策的回去问用户；可选的仅提示）
   ▼
[阶段5] 收敛为卡片（v1，排序改用"来源"而非编造的 confidence）
   │
   ▼
用户采纳/回答 → 回写 IR → 代码生成（demo/模块）
   │
   ▼
[阶段6｜新E] 生成后第二遍补全 ──► 渲染后才暴露的缺口（按钮没反应/流程断/状态没处理）
             （并入 ADR-0005 测试门/建造日志，产出新卡片，回到"用户采纳"）
```

**召回率优先级（修订 v1 第7节）**：① **IR 完备性批判 + 原型库覆盖度** ＞ ② 完善分类法 ＞ ③ 生成后第二遍 ＞ ④ 增加角色 ＞ ⑤ 提示词措辞。把精力投在 ①——它堵的是天花板。

## 1. 升级 A：IR 完备性批判器 + 30题补集锚定

**为什么**：v1 全部阶段扫的都是 IR；阶段0 抽 IR 本身是 LLM，**漏掉的整块实体/流程，后续全继承**。清单条数再多也扫不出 IR 里没有的东西。**真正的召回天花板在这里。**

**做法**：在规则派生前插一步，不查 IR 内部细节，只查"整块漏没漏"。两个锚:① 原型典型构成（impliedEntities/典型流程/典型角色）② **30题已覆盖维度清单的补集**（系统化扫这30题结构上没问到的维度）。

### 配置：30题覆盖维度清单（平台常量，与 checklist/archetype 并列维护）
```json
{
  "coveredDimensions": ["实体与字段", "核心页面", "主流程", "角色与可见范围", "产品定位"],
  "commonlyUncoveredDimensions": [
    "对账/统计/报表", "操作审计与留痕", "通知/提醒/消息", "导入/导出/批量",
    "多端/响应式", "离线/弱网", "数据归档/清理/配额", "对外接口/集成",
    "并发/重复提交", "权限的数据维度(非菜单维度)", "首次使用/引导/空数据",
    "异常分支与回滚", "搜索/筛选/排序的组合"
  ]
}
```
> `coveredDimensions` 随你的30题实际覆盖面填；`commonlyUncoveredDimensions` 是"这类应用常漏的维度"，作为补集扫描的靶子。

### 提示词（阶段0.5）
```
# 角色
你是资深B端解决方案架构师。你的任务**不是**检查 IR 内部的字段级缺口，而是判断
**IR 本身是否漏掉了整类实体/页面/流程/角色/维度**——方案里压根没出现、但这类应用几乎一定需要的东西。

# 输入
IR：{{ir_json}}
原型典型构成（该原型通常有的实体/流程/角色）：{{archetype_implied}}
30题已覆盖维度：{{covered_dimensions}}
常被遗漏维度（靶子）：{{commonly_uncovered_dimensions}}

# 任务
1. 对照原型典型构成，列出 IR 缺失的**整块**实体/页面/流程/角色（非字段级）。
2. 对照"已覆盖维度"取补集：在"常被遗漏维度"里，列出本类应用关键、但 IR 与30题都没触及的维度。
3. 每条说明：缺什么 / 为什么这类应用通常需要 / 用户若确实不需要可忽略。

# 约束
- 只提"整类缺失"，不重复 IR 已有实体的字段级问题。
- 宁少而准：每条须能对应"原型规律"或"未覆盖维度"，不凭空发明。
- 不发明用户明显不需要的功能。

# 输出（严格JSON数组，无任何额外文字）
[{ "kind":"entity|screen|flow|role|dimension", "missing":"跟进记录实体",
   "why":"CRM核心是跟进过程，几乎必有", "ignorableIf":"纯静态客户名录则不需要",
   "source":"archetype|uncovered-dimension" }]
```
> 命中的"整块缺失"应回流补进 IR（新增实体/页面/流程占位），再进入后续阶段——否则后面照样扫不到它的细节缺口。

## 2. 升级 C：按"来源"定信任（弃用 LLM 编造的 confidence）

**为什么**：v1 的 0.85/0.9 是 LLM 或硬编码编的数字，未经校准，拿它做默认勾选不可靠（同"代理信号弱"之病）。

**做法**：每条 gap 不打 confidence，打 **`source`（来源）**，信任度由来源决定（确定性来源 ＞ LLM 来源）：

| source | 含义 | 默认处置 |
|---|---|---|
| `derived` | 阶段1 规则派生的硬必备 | 默认采纳（高信任） |
| `archetype` | 原型库 impliedRequirements 命中 | 默认采纳 |
| `ruoyi-standard` | 若依标配能力（导出/字典/数据权限…） | 默认采纳 |
| `checklist-llm` | 阶段2 清单扫描，模型判定 | 默认不勾，人审 |
| `role-llm` | 阶段3 角色评审，模型判定 | 默认不勾，人审 |

阶段5 排序与默认勾选改为：**先按 source 信任分组（derived/archetype/ruoyi-standard 在前、默认勾选），再按 category 分组**。confidence 字段保留供参考但不作决策依据。

## 3. 升级 D：处置分类（autofill / ask / info）

**为什么**：v1 把所有缺口都转成"采纳/忽略卡片"。但有些缺口是**业务决策**（销售能否互看客户？金额精度？状态机谁可改？要不要审批？）——**这些该回去问用户，不该替他猜一个默认**。混在一起既不准、又把用户淹没。

**做法**：在阶段4.5（一致性校验后）对每条 gap 判处置方式：

### 提示词（阶段4.5）
```
# 任务
对每条补全建议，判定其处置方式 disposition：
- "autofill"：有业界默认、低风险（空状态/加载态/删除二次确认/字段级校验/分页）→ 直接补默认，无需问用户。
- "ask"：属业务决策、无通用默认、答错代价高（数据权限范围/状态机谁可改/是否需审批流/金额精度规则/谁能删）
         → 生成一个给非技术用户的二选一(或单选)小问题，**不替他猜默认**。
- "info"：可选增值（如导出、批量导入）→ 仅提示，默认不做。

# 输出：在每条 gap 上加字段
{ ...原gap, "disposition":"autofill|ask|info",
  "question":"销售之间能互相看客户吗？",          // 仅 ask 给
  "options":["只看自己负责的","看本部门全部","都能看"] } // 仅 ask 给
```
> 前端据此分流：`autofill` 进采纳卡片（默认勾）；`ask` 进**追加问答**（补在30题之后的精准追问，最高价值）；`info` 折叠在"可选增强"里。这把"补得多"变成"补得准"。

## 4. 升级 E：两遍补全（生成前 + 生成后），并入 ADR-0005

**为什么**：v1 是生成**前**一次性补全。但很多缺口要**看到渲染出的界面**才暴露——按钮没反应、流程断头、某状态没处理。IR 扫描看不见，渲染后能看见（ADR-0005 的"截图检查"洞见）。

**做法**：补全从"一遍"改"两遍"：
- **前置遍**：本工具包（阶段0–5）→ 回写 IR → 生成 demo/模块。
- **后置遍**：demo/模块**生成并渲染后**，复用 [ADR-0005] 的建造回路——`BuildJournal` 的"还可能漏了啥"批判 + 测试门发现的"功能界面缺可操作元素/流程断"，**产出新一批 gap 卡片**，回到"用户采纳"，再触发该模块重建（建造回路本就可逐模块续跑）。

> 这是 v1 与 ADR-0005 的合流点：**需求补全不是一次性流水线，是建造回路的一环**。前置补 IR 推得出的；后置补 artifact 才暴露的。

---

## 5. 升级 F（轻）：完备性自批 + 召回反馈回路

- 末尾加一个"还可能漏了啥"的完备性批判器（一次低成本 LLM），对剩余 gap 集做"这类应用是否还有未覆盖的大类"自检。
- **召回靠长大、不靠断言**：每个真实交付项目事后暴露的遗漏，回流去扩 `archetype` impliedRequirements / `checklist` / `commonlyUncoveredDimensions`。v1 说升级优先级是"①扩原型②完善分类法"，但没说"从哪来"——答案是**从真实遗漏来**。

## 7. 接进平台

- **位置**：本工具包是平台「需求→规格」阶段的内核，夹在 idea-interview（30题）与 demo/模块生成之间。
- **IR ↔ Specification**：IR 即结构化规格，落到平台现有 `Specification`（含 acceptanceScenarios、资料 provenance）的扩展；回写后的 IR 是代码生成的最终输入。
- **与 ADR-0005 合流**：升级 E 的"后置遍"复用建造回路的测试门/BuildJournal；`ask` 类追问可走平台已有的人在回路（awaiting_decision）。
- **常量化**：checklist / 各原型模板 / ruoyiCapabilities / coveredDimensions 作平台配置表，提示词以变量注入，独立升级。

## 8. 落地优先级（MVP）

1. **先做 A（IR 完备性批判 + 30题补集）+ v1 的 0/1/2/5** —— 召回天花板 + 硬必备，收益最大。
2. 加 **D（处置分类）** —— 把"补得多"变"补得准"，体感提升最大。
3. 加 **C（按来源定信任）** —— 默认勾选可信。
4. 加 **E（后置遍）** —— 与 ADR-0005 建造回路成熟后合流。
5. F（反馈回路）随真实项目积累再上。
