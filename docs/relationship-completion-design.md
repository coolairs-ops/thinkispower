# 实体关系补全设计（A/D 关系版 + 追加问答）

> 关系：扩展 [requirement-completion-kit-v2.md] 的 A/D/回写到「实体关系」这个新缺口类型；
> 喂 [ADR-0003] 若依**子表 codegen** 的输入（Phase 2 最现实的缺口）；复用 D 的 `ask`→追加问答。
> 状态：设计草案（Phase 2）。Phase 1（扁平实体 CRUD）不依赖本设计。

---

## 0. 动机

- 真客户系统（CRM/台账）核心是**实体之间的关系**（客户 1—N 项目、订单 N—1 客户…），但：
  - 扁平 IR / 当前 `ParsedModel` **不捕获关系**——这是 ADR-0003 Phase 2 的真缺口。
  - demo 物理上修不出"级联删除/关联列表"（自迭代诊断已证）；若依**子表 codegen** 才是落点。
- **不能指望非技术客户画 ER 图**。但客户**已经填的需求+设计里藏着关系线索**——把线索变成**几道选择题**让客户确认，比让他建模可行得多。

**一句话**：实体关系 = 需求补全流水线里的一种新缺口类型；检测候选 → 选择题问清 → 回写 IR → 喂若依子表。

---

## 1. 流水线（接进现有 A/D/回写）

```
实体 + 页面 + 功能 + 设计建议 + 流程（客户已填）
   │
   ▼
[检测] 候选关系扫描（A 的角色）──► 每条候选带「证据 + 来源」
   │
   ▼
[分类] 明显的 autofill / 模糊的 ask（D 的角色）
   │        autofill：客户-项目明显 1—N → 直接默认
   │        ask：基数模糊 / 是否多对多 / 级联行为 → 出选择题
   ▼
[追加问答] 选择题给非技术客户（D 的 ask 落点，新对话窗口）
   │
   ▼
[回写] 答案 → IR.relations ──► 若依子表 codegen / demo / 规格 全受益
```

---

## 2. 候选关系检测（核心：从已填的东西推出该问什么）

不穷举每对实体（N² 会淹死人），只对**有证据**的候选出题。证据源：

| 信号（客户已填） | 暗示 | 例 |
|---|---|---|
| **字段/外键**：子实体含 `客户`/`customerId` 字段 | 项目 N—1 客户 | |
| **页面结构**：`客户详情`页写"关联项目列表(含金额/进展)" | 客户 1—N 项目 | becca759 即此形 |
| **设计导航**：`客户→客户详情→项目详情 三级跳转` | 客户→项目 层级 | |
| **流程**：`给客户分配项目` | 客户-项目 关联 | |
| **实体描述**：`客户（含关联项目）` | 同上 | |

启发式抓出候选 → LLM 复核并补漏，每条候选输出：`{parent, child, 推测基数, 证据, source}`。**有证据，问出来才像好的需求分析师在问，不是瞎猜。**

---

## 3. 问什么（选择题，非技术友好，对齐若依 codegen 需要）

三类，正好映射若依子表 + FK：

| 题型 | 选择题（举例） | 映射 |
|---|---|---|
| **基数** | "一个【客户】下面可以有多个【项目】吗？" → `[能,多个 / 不能,一对一 / 它俩没关系]` | 1—N / 1—1 / 无 → 若依子表 |
| **必填** | "【项目】一定要属于某个【客户】吗？" → `[必须有 / 可以先没有]` | FK 是否可空 |
| **级联**（业务决策，**必问**） | "删除【客户】时，名下项目怎么办？" → `[一起删 / 保留但解除关联 / 不许删有项目的客户]` | onDelete: cascade/setNull/restrict |

> 级联是"答错代价高的业务决策"，正是 D 的 `ask` 该管的，**绝不替客户猜默认**——也正是 demo 修不出那块的真正问法。

---

## 4. 设计纪律（免得问烦客户）

- **只问有证据的候选**，不穷举（宁少而准，承接 kit 的召回纪律）。
- **明显的 autofill、模糊的才 ask**：客户-项目明显 1—N 就默认；只把"是否多对多/可不可选"这种模糊的拎出来问。
- **级联永远问**：业务决策，不猜。
- **每条带 `source`**（承接 kit 升级C 按来源定信任）：字段/页面证据强→可默认；纯 LLM 推测→必问。

## 5. IR 表示（回写落点）

在 `structuredRequirement` 加 `relations`（与 `completenessGaps` 并列）：

```json
{
  "relations": [
    { "parent": "客户", "child": "项目", "cardinality": "1-N",
      "fkField": "customerId", "required": true,
      "onDelete": "restrict", "source": "page", "confirmed": true }
  ]
}
```

- `cardinality + fkField + required` → 若依 **gen_table 子表配置**（subTableName/subTableFkName）。
- `onDelete` → FK 约束 / 业务规则（restrict=不许删、cascade=一起删、setNull=解除）。
- `confirmed`：autofill 默认为 true；ask 经客户答后置 true。
- 回写后**下游全受益**：若依子表 codegen、demo 生成（关联列表/级联提示）、规格（数据模型关系）。

## 6. 关系类型分阶段（诚实的难度）

- **Phase 2a｜1—N 主子表**（✅ 已落）：若依子表 codegen 直接支持，最常见（覆盖绝大多数台账/CRM）。
- **Phase 2c｜自关联/树**（✅ 已落，部门树、分类树）：`parent===child`（或显式 `tree:true`）即判树 → 同实体补可空自外键(parentId) → 若依**原生 tree 模板**（`tplCategory:'tree'` + treeCode=pk / treeParentCode=自外键 / treeName=显示名列）。树优先于 sub 判定，避免自嵌套被误判成主子表。
- **Phase 2b｜N—N 多对多**（✅ 已落，识别+合成中间表）：检测互为多 → `cardinality:'N-N'`+`joinTable` → `synthesizeJoinEntities` 合成中间表实体（自增主键 + 两端 BigInt 外键），DDL/gen_table 自动产出、两端表保持 crud、双向可查。**跨表 master-detail UI 联动**（若依 codegen "更绕"那块）随 M3c-remaining（Velocity 坑已停）后续做。

---

## 7. 触发与 UX（追加问答窗口）

- **位置**：D 的 `ask` 落点"追加问答"——**主初访(30题)之后、生成之前**的聚焦二次问答窗口。
- **触发**：**检测到模糊候选才弹**（非每次、非穷举），且只弹 ask 类的几道。不打扰。
- **形态**：选择题为主（基数/必填/级联），一问一答或一屏几题。**新对话窗口**承载，与主访谈解耦。
- 客户也可主动进入（"完善实体关系"入口）作兜底。

---

## 8. 与现有的衔接（全是复用，不重造）

- **A** → 候选关系检测（gap 的角色从"缺实体/页面"扩到"缺关系"）。
- **D** → autofill/ask 分类 + 选择题生成（已有 question/options 机制，直接用）。
- **回写** → `apply` 扩一类：`relation` 缺口回写 `structuredRequirement.relations`（类比 screen→pages）。
- **C（按来源定信任）** → 关系候选的 `source` 定默认/必问。
- **ADR-0003** → relations → 若依子表 codegen 的输入（Phase 2 缺口的解）。
- **通用性**：回写进 IR 后不只为若依——demo/规格同样吃到，是平台级能力。

---

## 9. 落地优先级（MVP）

1. **检测 + 1—N + 选择题 + 回写**（Phase 2a）✅：扫候选→出基数/级联题→答→回写 relations。最小闭环、覆盖最广。
2. 接若依子表 codegen（消费 relations）——与 ADR-0003 M3c-remaining 合流。
3. N—N / 树（Phase 2b/2c）✅ 检测+回写+codegen 输入全打通（中间表/树模板）；live 验证随 M3c-remaining。
4. 主动入口 + 追加问答窗口 UI（✅ 追加问答合批窗口已落，树/N-N 的 ask 题复用同一窗口零改动）。

> Phase 1（扁平实体 CRUD 真出若依）**不等本设计**；本设计是把"能出"升级为"出得对（带关系）"。
