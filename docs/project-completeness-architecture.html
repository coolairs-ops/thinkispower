需核对

运行拓扑和仓库文件存在不一致

docker-compose.yml、README 与文档都引用 internal/cc-bridge，但当前仓库文件清单没有该目录；如果这不是子模块或部署环境注入，会导致 compose 构建 cc-bridge 失败。

需验证

多租户模型已落库，但仍要确认所有查询都按 orgId 收口

Prisma 已有 orgId 和 membership，但完整性不只看 schema；项目、导入、交付、守护、经验推荐等控制器和服务都需要统一租户边界测试。

优势

测试资产很厚，质量闭环不是纸面设计

仓库包含大量 API 单测、集成/e2e 测试和前端 Playwright 测试，覆盖认证、项目流、Demo、反馈、交付、若依运行时、规则引擎、知识、守护、交付门等关键区域。

六、完整性缺口与优先级

P0 运行拓扑补齐

  • 确认 internal/cc-bridge 是否遗失、外部注入或已被新模块替代。
  • 若已替代，更新 docker-compose.yml、README、n8n workflow 与代码中的硬编码 URL。
  • 用一次干净环境 docker compose build 验证一键启动链路。

P1 租户隔离闭环

  • 以 orgId 为主线补全项目、导入、交付、守护、经验推荐的查询约束。
  • 给跨租户读取、更新、导出、预览访问补 e2e 反例。
  • 明确个人项目和组织项目的产品规则。

P1 异步任务可靠性

  • Demo、交付、自迭代、守护巡检都应以持久队列和数据库状态为事实源。
  • 统一失败重试、超时、取消、恢复和前端进度查询。
  • 减少进程内事件/SSE 内存状态对结果正确性的影响。

P2 文档校准

  • 旧文档里“19 个模型、无多租户”等描述已经过期。
  • 建议保留一份“当前真相源”架构页，其他文档链接到它。
  • 把端口 3001/3003/5000/5001 与本地脚本统一说明。

P2 外部依赖边界

  • DeepSeek、Qwen、MinIO、若依、n8n、Docker socket 都应有降级或健康检查。
  • 交付失败应能映射为用户可理解的状态，而不是暴露技术细节。
  • 外呼、密钥和成本配额建议集中到 LLM Gateway 与租户配额。

P2 质量指标产品化

  • 已有 L1/L2/L3、Guardian、Go-live Gate，适合沉淀成统一质量报告。
  • 把传感器证据、修复记录、回滚点串成用户可读的验收凭证。
  • 经验推荐与案例复盘可反哺新项目生成。
