## 工程控制论方法融合

本平台的核心设计哲学借鉴了**工程控制论**的基本原理，将 AI 代码生成从不稳定的"一次性输出"改造为可控、可观测、可收敛的"闭环系统工程"。

### 1. 反馈闭环 (Feedback Loop)

```
设定目标 (需求)
    │
    ▼
执行动作 (AI生成) ──▶ 输出结果 ──▶ 传感器测量 ──▶ 偏差计算 ──▶ 纠正动作
    ▲                                                              │
    └──────────────────────────────────────────────────────────────┘
                        闭环迭代 (最大收敛)
```

**传统 AI 生成**: 开环 — 输入→输出, 不管结果对不对, 一次性交付。  
**控制论方法**: 闭环 — 生成→测量→比较→纠正→再生成, 直到偏差可接受。

### 2. 传感器系统 (多维度感知)

类比工业控制中的多传感器融合（温度+压力+流量+振动），平台采用 4 层传感器：

| 传感器 | 检测维度 | 控制论等价 |
|--------|---------|-----------|
| **L1 静态分析** | HTML结构/标签完整性/代码规范 | 静态特性测量 |
| **L2 运行时** | 交互响应/导航可达/状态变量 | 动态特性测量 |
| **L3 语义理解** | 功能覆盖/需求追溯/逻辑合理 | 语义特征提取 |
| **CrossValidator** | Qwen独立交叉验证 | 冗余传感器校验 |
| **TraceabilityValidator** | 需求→功能→代码追溯 | 特征匹配+信息融合 |

```
SensorFusion(加权融合) = L1×0.25 + L2×0.25 + L3×0.25 + CrossValidator×0.15 + Traceability×0.10
```

### 3. 决策引擎 (状态机 + 阈值控制)

类比 PID 控制器和状态机控制，决策引擎根据 6 维度完整度和置信度计算下一步建议：

```
完整度 = 描述×15% + 方案×20% + 功能×20% + 页面×15% + PRD×20% + 规格×10%
置信度 = 已填写维度 × 15 + 10 (基数)

决策规则 (9条, 优先级排序):
  错误 > 5                    → needs_human   (紧急停机)
  高风险 ≥ 3 + 预算 < 1000    → pause         (保护性停机)
  完整度 < 30%                → continue_clarify (继续采集)
  周期 > 60天 + 功能 > 5      → narrow_scope  (参数约束)
  plan_ready + 无规格         → generate_spec (状态转移)
  spec_confirmed              → generate_demo (状态转移)
```

**控制论映射**: 状态机 = 系统状态空间; 决策规则 = 控制律; 完整度 = 状态向量; 置信度 = 观测可靠性。

### 4. 自愈流水线 (故障检测 + 自动修复)

类比工业中的容错控制和自恢复系统，4道闸门构成质量防火墙：

```
生成输出
  │
  ├─ 闸门1: 结构检测 → ──不合格──▶ 自动重试(temperature+0.1)
  │    合格 ↓
  ├─ 闸门2: 内容检测 → ──不合格──▶ 清洗+强化Prompt
  │    合格 ↓
  ├─ 闸门3: 隔离检测 → ──不合格──▶ 快照回退
  │    合格 ↓
  └─ 闸门4: 验收检测 → ──不合格──▶ 标准化替换(Dockerfile等)
       合格 ↓
      交付输出
```

**重试策略**: 第1次原Prompt(t+0.1) → 第2次强化Prompt → 第3次降级/返回null。
**控制论映射**: 闸门 = 参考输入比较器; 重试 = 控制系统鲁棒性; 降级 = graceful degradation。

### 5. 三级降级 (鲁棒控制)

```
主路径: 分步生成   (精确控制, 4步独立反馈)
  ↓ 失败
备用1: CC Bridge   (模型切换, Claude Code)
  ↓ 失败
备用2: Cloudecode  (简化控制, 单次生成)
  ↓ 失败
安全态: build_failed (安全停机, 通知用户)
```

**控制论映射**: 冗余执行器; 三级 = 硬件冗余; 安全态 = fail-safe 状态。

### 6. 复盘校准 (模型自适应)

类比自适应控制中的参数辨识，每次交付后对比预测与实际：

```
预测成本/周期 ──▶ 实际成本/周期 ──▶ 偏差计算 ──▶ 校准因子 ──▶ 下次预测修正
```

**控制论映射**: 系统辨识 → 传递函数修正 → 模型自适应。

## 五、关键技术决策

### 5.1 为什么分步生成而不是一次生成

AI 单次生成全栈代码的质量不可控。分步生成让每步独立 prompt, 上一步输出作为下一步上下文, 显著提高代码结构质量。

### 5.2 为什么标准化 Dockerfile

AI 生成的 Dockerfile 经常使用 pnpm/yarn, 包含 prisma generate 等不稳定步骤, build target 不匹配。标准化为 npm + 多入口容错方案, 部署成功率从 0% 提升到可验证。

### 5.3 为什么三级降级

分步生成最完整但最慢, CC Bridge 中等, Cloudecode 最快。三级降级确保交付不会因单一引擎失败而中断。

### 5.4 为什么 token → accessToken

P5 安全加固引入 JWT 双 token 机制后, 前端字段名未同步导致静默登录失败。教训: API 响应格式变更必须双端同步。
