# 思想动力价值与实现白皮书 · v2（校准整合版）

**文档定位**：产品宪法。所有功能、技术选型、交互打磨对齐本文价值承诺。
**与 v1 的区别**：① 改为**单客户单系统**模型（无多租户）；② 每条价值标**兑现度**（✅已兑现 / 🟡有地基 / 🔴待建），守"靠实测不靠自评"；③ 把全自然语言需求交互设计织进价值一/六；④ 校准被夸大的承诺，划出诚实边界；⑤ 修正路线。
**对象**：政企信息中心主任、业务负责人、决策层。

---

## 一、交付模型（v2 校准：单客户单系统）

**一个客户 = 一套独立私有化部署的若依底座。** 同一客户内业务人员"种"出的多个应用（如客户管理、设备报修、巡检）**共用这一套底座**，靠角色 + 数据权限 + 菜单分隔。**不涉及多租户**——客户之间天然物理隔离（各自部署）。

> 这把"多租户隔离"从待办里移除：客户间隔离 = 各自独立部署；客户内多应用隔离 = 角色/菜单作用域（已实现）。

---

## 二、客户的四重困境（不变，真实痛点）

| 困境 | 表现 |
|---|---|
| 需求碎片化 vs IT 稀缺 | 80% 需求是增删改查小改，却 100% 要 IT 排期 |
| 合规高压 vs 业务灵活 | 监管趋严，业务为效率用 Excel/个人 SaaS 绕监管 → 影子 IT |
| 系统烟囱 vs 数据统一 | 几十套系统数据不通，同一"门店"各系统定义不一 |
| 供应商锁定 vs 自主可控 | 核心系统被绑定，改一下高价 + 漫长排期 |

---

## 三、六大价值 × 兑现度校准（核心）

### 价值一：需求响应从"周级"到"分钟级" 🟡（分档兑现）
- ✅ **配置/权限/菜单类改动**：写库即生效，**真分钟级**（若依运行时配置，不重启）。
- 🟡 **新增模块/改业务逻辑**：要重编译 + 重启若依实例，**分钟级停服窗口**（冷启 1.5–11min，Windows 文件桥所致）。单客户私有化下只影响该客户、可错峰。
- **诚实表述**：对外说"配置级改动当天生效；新增模块按天级、且全程留痕审核"，**不说"1000 倍/统一 15 分钟"**——分档讲，别立兑现不了的 flag。
- **引擎 = 全自然语言需求交互**（见第四节），它决定"需求→规格"这段快不快、准不准。

### 价值二：统一底座治理，收编影子 IT 🟡（收窄口径）
- ✅ **真兑现**：平台**种出来的应用**全跑在统一底座（统一认证/权限/审计/备份），新模块数据天然合底座标准。
- 🔴 **未兑现（v1 写超了）**："收编全公司所有数据 / 统一数据资产池"——这是**全域数据中台愿景，已封存**（见 `VISION-PARKING`）。现在**不承诺**"把所有影子 IT 和外部数据都收进来"。
- **诚实表述**："让新建的应用从源头就在治理框架内"，不是"一夜收编所有历史数据"。

### 价值三：全链路可审计 ✅（已兑现，最强项）
- ✅ 每次对话修改/AI 决策/代码生成/测试/人工审核 = 结构化日志；上线门据实留痕（goLiveStatus）。这是平台的**真护城河**，也最该对政企讲。
- 待补：一键导出"任意时段/模块完整审计报告"的成品页（地基有，报表页要做）。

### 价值四：连接既有系统、打通孤岛 🔴（蓝图，未起）
- 🔴 **连接器运行时 = 未建**（能力清单 🔴）。当前对 ERP/OA 对接**没有现成能力**。
- **诚实表述**：这是**路线图能力，不是已交付**。对客户讲要明说"规划中/按需定制"，不能当现货卖。对私有接口可先产《接口依赖待办清单》（这个生成能力可做）。

### 价值五：自主可控、源码交付 ✅（已兑现）
- ✅ 私有化部署、底座（若依 MIT + 平台代码）交付客户服务器、数据不出域（`AI_MODE=local`）。这条对政企/信创是**真卖点且已成立**。

### 价值六：重塑 IT 角色 ✅（机制已成立）
- ✅ 业务自助"种"应用、IT 转向审核/治理——脊柱已证（4 样板零手工跑通）。这条随价值一的需求交互一起落地。

**一句话校准**：**真能打的是价值三（可审计）+ 价值五（自主可控）+ 价值六（角色重塑）+ 价值一的配置档**；**价值二要收窄、价值四是蓝图**——对客户讲清"现货 vs 路线图"，是守可信的底线。

## 四、全自然语言需求交互设计（价值一/六的引擎）

**目标**：消灭"需求反复"——把返工从昂贵的交付后，前移到便宜的澄清阶段。
**特点决定形态**：用户非技术 + 交付目标形状已知（实体/字段/关系/角色/数据范围/菜单/验收场景）→ 交互不是开放对话，是**填已知槽位的覆盖式收敛**。

```
① 大白话进（说一句 / 传文档 / 传截图）
② 抽成结构化 IR，对齐到若依交付的固定槽位（目标形状已知 → 澄清不开放）
③ 覆盖式缺口扫描（完备性批判 A + 覆盖度量化）→ 缺口带证据
④ 缺口 → 一屏业务选择题（处置分类 D：autofill/ask/info）
   ★ 顶部"需求完备度 72% → 还差 3 个关键决策"进度条（让业务人当场看见差什么，一次问清）
⑤ 每个回答写进规格"澄清记录"区（可追溯：这字段为什么这么定 = 你哪天选的）
⑥ 试切确认：给可点 demo 确认逻辑，不让业务人读规格
⑦ 冻结 = proposal→apply 状态机；要改 = 新 proposal，版本化（接"交付后变更"）
```

**融合自三个高星成熟方案（借模式、不装工具、不照搬技术问答表面）**：
- **GitHub Spec-Kit `/clarify`**（55k★，覆盖式提问 + Clarifications 记录）→ 升级现有"完备性批判 + 追加问答"为**完备度进度条 + 澄清记录区**。
- **BMAD-METHOD**（48k★，PM/架构师 persona）→ 作为 ADR-0014"业务翻译官/领域建模师"提示词蓝本。
- **OpenSpec**（52k★，proposal/apply/archive 状态机）→ 规格冻结/解冻/再交付版本化。

**实现纪律**：长在现有 discovery 上（加进度条 + 澄清记录 + 冻结状态机三样），不重做；表面是业务选择题 + demo，不是技术问答；它只管"需求→规格"，"功能对不对"交 ADR-0015 终态验收。

## 六、产品设计铁律（保留，已对齐现有门）

1. **AI 生成代码必须在底座内运行**，不绕权限/审计/日志 → 对齐若依底座 + SecurityGate。
2. **任何变更必须过测试，不过自动回滚** → 对齐上线门（ADR-0009 确定性二值闸）+ 守护回滚。
3. **所有操作留痕** → 已有结构化日志 + goLiveStatus。
4. **底座源码开源交付，客户 100% 所有权** → 已成立（价值五）。
5. **对话唯一入口，信息深度渐进**（业务看结果、IT 看过程）→ 对齐双模式 + 透明度分级。

> 新增铁律建议：**6. 对外承诺只说"现货"，"路线图能力"必须标注"规划中"。**（守可信，防价值二/四被当现货卖。）

## 七、校准后的路线（从需求目标倒逼）

**当前主线（唯一闸）：serve/部署基建自动化** —— 让 productionUrl 由真部署产出（CI/容器/进程托管替代手工 vite preview）。这是脊柱从"我机器上跑通"到"能真交付给一个客户"的最后一道闸（多租户闸已被单客户模型消解）。

**然后（按价值倒逼排序）**：
1. **价值一引擎**：全自然语言需求交互（第四节）+ ADR-0014 业务翻译官角色 → 把"需求→规格"做快做准。
2. **价值三成品**：一键审计报告导出页（地基有，补成品）。
3. **价值四/二（蓝图转地基）**：连接器运行时 + 行业知识库 → 进 Phase 3 行业根据地，**有真客户需求再建，别预建**。
4. 留口解冻：ADR-0013 口一（dataScope 中立化，样板已稳可排期）、ADR-0015 G3（前端终态断言）。

**封存不做**：全域数据中台（价值二的过度部分）、大屏/C 端/非 Java（ADR-0013 留口未盖楼）。

---

## 八、诚实边界（对客户怎么说）

- **能现货交付**：私有化单客户若依控制台、可登录、角色数据权限、全程审计留痕、源码交付、配置级改动当天生效。
- **按天级/审核交付**：新增模块（重编译重启窗口）。
- **路线图/按需定制（不当现货）**：ERP 连接器、行业知识库、全域数据治理。
- **不做**：多租户 SaaS、对外门户/大屏/C 端（当前路线外）。

**总结**：思想动力对政企的真承诺 = **一套私有化、可审计、自主可控、业务能自助改的若依统一控制台**——这是已兑现的硬货；连接器与行业库是路线图，全域数据是远期愿景。守住"现货 vs 蓝图"的诚实分界，就是守住政企最看重的可信。
