## 三、数据可信：知识库如何支撑评分（knowledge.schema.json）

**承诺**：评分权重可由客户主观设定（formula 的 weighted_sum，权重值客户配+留痕），但**每个维度的取值必须可溯源到知识库里的真实材料**，不能是 AI 编的或拍脑袋的。

**知识库三层 + 一条溯源链：**
```
维度得分 → metric值 → confirmed的Fact(结构化值) → Evidence(原件中的原文片段) → Source(哈希校验未篡改的原件)
```
- Source(原件层)：原件只读、留哈希、永不物理删除(只标失效)。
- Evidence(证据片段层)：原件中被用到的某一处=原文+精确位置+指回原件。**这是 rulepack 里 evidence_ref 真正指向的东西**(原本空ID，现在落地)。
- Fact(结构化值层)：进评分的值，**每个必须挂≥1条 evidence**。只有 status=confirmed 的 Fact 才能被 metric 取数聚合。

**提取流程四步（核心原则：AI找候选、机器卡假出处、人确认采纳、缺失显式）：**
1. AI提取**强制附原文+位置**，给不出出处的候选作废（掐死幻觉）。
2. 机器校验门：回原件核对原文是否真存在，对不上作废。
3. 人工确认：过门的是 candidate，人确认才变 confirmed、才可进评分（复用 evidence_policy 的"待人工确认"）。
4. 缺失显式：维度需要但无材料支撑的，标 status=missing，**绝不静默补0或给默认值**。

**与规则引擎的接法**：本质是把 evidence_policy 从"结论级"下沉到"数据级"——以前是"结论要有依据"，现在连"每个参与计算的数据"都要有依据。metric 的数据源从"假设字段已有数"改为"绑定 confirmed 的 Fact 取数"；rulepack 的 evidence_ref 正式指向 knowledge 的 evidence_id。守护接入：原件哈希变了(被篡改)、某 Fact 的支撑证据被作废(支撑没了)→ 报警。

> ★ 诚实提醒（写给产品也写给工程）：逐条人工确认是**本质成本**，不是设计缺陷——"每个数字可信溯源"与"全自动零人工"在监管场景不可兼得。故 knowledge.schema 的提取严格度**做成可调档**(逐条/批量/抽样/高可信自动采纳)，药监等强溯源场景用严档，轻量场景可放松。**不要一刀切全严。**

## 四、规则引擎放在平台哪个环节，以及为什么这么放

平台主链路：① 需求采集 → ② 设计确认 → ③ 真后端生成 → ④ 交付上线 → ⑤ 持续守护。

**规则引擎不是放在某一个环节的功能，它有两个形态、放在两个位置：**

### 形态A：配置态 —— 放在 ① 需求采集的延伸（①→②之间新增"规则定义"子环节）

**放这里的原因：**
1. "业务专家配规则"这个动作，发生在系统**还没上线、正在被定义**时——和"补全需求、确认设计"是同一阶段的事，本质就是在回答"这系统按什么逻辑判断"，属于需求的一部分。
2. 可**直接复用①已有的能力**：①已经有"结构化访谈+完备性批判+把问题合成一屏选择题"的成熟机制。配规则走同一套交互——把"这条规则阈值多少""这个指标怎么从数据算"做成同样的选择题/填空。**规则配置=需求采集的一个专门化分支**，不用另造交互。
3. 配完的规则要在 ② 设计确认时**即时试算**给业务专家看（"按你配的规则，这家企业算出来D级86分，对不对？"）——试算正好落在②"让人点着Demo确认"这一步。

> ★ 即时试算是配置界面的命根子（抄Excel：改一个值结果立刻变，一眼知道对没对）。没有即时试算，业务专家配三条就放弃。**必须做。**

### 形态B：运行态 —— 放在 ③ 生成的后端里，作为 BackendRuntime 的标准内置服务

**放这里的原因：**
1. 配好的规则包要被③生成的后端**加载执行**：用户每查一个对象（一家企业/一台设备），后端读规则包、拿真实数据、跑八步、出结论。这发生在④上线后的**运行时**。
2. 关键决定——**做成 BackendRuntime 的一个标准内置服务，不是每个项目现写的代码**。即：任何被生成的系统天生带一个"规则执行引擎"插槽，加载对应行业规则包就能跑。
   - 这样规则就从"人手写的代码"变成"引擎读取的**配置数据**"。
   - 配置数据 → ⑤守护能**天然覆盖**（改规则自动留痕、可回滚、可体检失效规则）。**这一步直接解决了"手写定制区守护够不着"的老问题。**

### 贯穿①③④⑤的一条线：证据与人工确认（evidence_policy）

引擎强制、非行业可配。每个结论必须回指证据、默认"待人工确认"、关键证据缺失标"待核实"绝不自动下结论。这是平台"可信底座"的信条，从配置态一路管到运行态。

### 一张图

```
① 需求采集 ──┬─→ [新增:规则定义子环节]   ← 配置态(复用①的选择题式采集)
             ↓
② 设计确认 ──── Demo里【即时试算】规则      ← 让业务专家点着确认算得对不对
             ↓
③ 真后端生成 ── 后端内置【规则执行引擎】     ← BackendRuntime标准服务，规则包=配置数据
             ↓
④ 交付上线 ──── 运行态:每查一个对象跑一次规则
             ↓
⑤ 持续守护 ──── 守护天然覆盖规则:留痕/回滚/体检失效规则
  └── 贯穿①③④⑤: 证据+人工确认 (evidence_policy, 引擎强制)
```

## 五、开关：规则引擎是"可选模块"，三条铁律

1. **开关绑在每个项目上，不是全局**。它就是①需求里的一道选择题"这系统要不要风险评分/自动定级？"（对应 schema 的 `meta.enabled`）。
2. **关掉=根本不生成规则产物**（无规则表、无规则服务、无规则入口），保持纯CRUD的轻量。**不是"生成了再隐藏"。**
3. 由此平台自然分两档：轻量档(纯CRUD) / 专业档(带规则)，同一套底座、同一道选择题切换。

## 六、v1 边界（哪些先做、哪些留v2）—— 详见 schema 的 _v1_scope_note

| 能力 | v1 | v2 |
|---|---|---|
| metric aggregate族(count/sum/avg/ratio/min/max/latest...) | ✅ 全做 | — |
| metric temporal族 | 只做 trend(升/降/平) | slope/yoy/mom/diff |
| as_of 时点计算 | 当前时点 + 记录所用规则版本 | 任意历史时点回溯重算 |
| formula | weighted_sum/normalize/product/piecewise | matrix_lookup(不规则矩阵) |

> 原则：**schema已留好v2全部位置，做v2不需要改地基**。v1聚焦"业务专家一填就对、一试就懂"的部分，别让高复杂度能力拖累引擎的简单可用。

## 七、动工顺序

1. ✅ **JSON Schema**（已完成，即 rulepack.schema.json）—— 配置界面与引擎共同的契约。
2. **配置界面**（形态A的落地）：把六个补丁做成"搭积木"式可视化表单 + **即时试算** + 行业模板加载。
3. **执行引擎**（形态B的落地）：按 schema 的八步执行流程实现 v1 范围，做成 BackendRuntime 标准服务。用六个 通用性验证_*.json 当测试用例验收。
4. **知识库与提取流程**（数据可信的落地）：按 knowledge.schema.json 实现 Source/Evidence/Fact 三层 + 四步提取。metric 取数改为绑定 confirmed 的 Fact。可与 2、3 并行设计，但 metric 取数接口需与引擎对齐。

> 2 和 3 都依赖 1，可在 1 完成后并行。
