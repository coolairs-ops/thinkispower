# 通用规则包标准 Schema v1.3（封版）

> 这是"任何行业规则包都必须长这样"的规范。
> 配置界面照它生成录入表单；执行引擎照它解析执行；守护照它做体检。三方共用一个地基。
> 已通过跨行业验证（药监风险画像 + 贷款审批），并据此补入三处关键能力（见末尾变更记录）。

---

## 顶层结构

```
RulePack {
  meta            // 元信息：名称、版本、所属项目、行业标签
  data_bindings   // 第一层：引用平台已生成的实体/字段（不重定义）
  metrics[]       // 第二层：指标定义
  formulas[]      // 第三层：公式/评分
  rules[]         // 第四层：触发规则
  conflict_policy // 【补丁2】冲突裁决策略（一等公民）
  evidence_policy // 贯穿线：证据与人工确认（引擎强制，非行业可配）
  // 执行时可带 as_of 时点参数【补丁6】：默认当前，可指定历史时点回溯选规则
}
```

---

## meta（元信息）

| 字段 | 类型 | 说明 |
|---|---|---|
| name | string | 规则包名称 |
| version | string | 版本号（每次改规则递增，配合守护做留痕/回滚） |
| project_id | string | 绑定的项目（开关绑在项目上，不是全局） |
| industry_tag | string | 行业标签（药监/信贷/核保…，仅用于分类检索） |
| enabled | bool | 【开关】此项目是否启用规则引擎。false=该项目根本不生成规则产物 |

---

## 第一层 data_bindings（数据绑定）

引用平台已生成的 Prisma 实体，**不重新定义数据结构**。

```
data_bindings: [
  { entity: "企业", fields: ["企业类型", "经营链条"] },
  { entity: "检查记录", fields: ["检查类型", "检查日期", "缺陷数"] }
]
```

---

## 第二层 metrics（指标）

```
Metric {
  id           // 唯一标识，供公式/规则引用，如 M_飞检次数
  label        // 人类可读名
  source_type  // 【补丁4】指标值的来源类型 —— 医疗验证逼出
  metric_family// 【补丁5】计算族：aggregate | temporal —— 电网验证逼出
  aggregation  // family=aggregate 时：count|sum|avg|ratio|min|max|latest|earliest|last_n
  temporal_op  // family=temporal 时：trend(升降平,v1) | slope|yoy|mom|diff(v2)
  source       // 作用的实体.字段
  filter       // 可选：过滤条件（含时间窗）
  manual_spec  // 当 source_type=manual 时用：{ 录入角色, 评分标准(可挂上传文档), 取值范围 }
  evidence_ref // 该指标回指的证据来源（贯穿线要求）
}
```

### 【补丁4】source_type（指标来源类型）—— 医疗验证逼出

现有 schema 原本暗含一个假设：指标都能"从数据算出来"。三甲医院评审打破了它——大量指标是**专家现场打分录入**的，没有"聚合"这个动作。故为 Metric 增加来源类型：

| source_type | 含义 | 行业例 |
|---|---|---|
| `computed` | 对数据做聚合算出 | 药监 飞检次数；贷款 逾期次数 |
| `manual` | 指定角色按标准人工录入/打分 | 医疗 条款评分；ESG 定性评级；职称评定 |
| `external` | 从外部系统取值（预留） | 集采/医保数据（呼应早期外部接口话题） |

`manual` 指标必须带 `manual_spec`（谁能录、按什么标准录、取值范围），且录入同样进"证据+人工确认"贯穿线。
**意义**：引擎从"只能处理算得出的硬指标"扩展到"能处理专家主观判断的软指标"，覆盖面从'数据驱动型评分'扩到'专家评议型评审'。

### 【补丁5】metric_family（指标计算族）—— 电网验证逼出

现有 schema 还暗含第二个假设：**指标 = 对一组值做聚合（塌缩成一个数）**。电网设备评估打破它——出现了"看一串时序值的变化趋势"这类指标（乙炔含量当前未超标，但近6个月持续上升即预警）。趋势不是聚合，是另一类计算。故引入计算族：

| metric_family | 含义 | 原语 | 行业例 |
|---|---|---|---|
| `aggregate` | 把一组值塌缩成一个数 | count/sum/avg/ratio/min/max/**latest**/earliest/last_n | 药监/贷款/医疗 全部指标 |
| `temporal` | 看一组时序值的变化形状 | 变化率/斜率(回归)/差分/同比/环比/趋势判定(升降平) | 电网趋势预警；设备运维；健康监测；经济指标 |

**分期落地（重要权衡）**：
- schema 现在就**预留 `metric_family` 维度和 temporal 族的位置**（地基对，以后不改架构）。
- 第一版引擎**只实做 aggregate 族 + 最简趋势判定**（近N期是否单调升/降），把回归斜率/同比环比标为 **v2 能力**。
- 理由：聚合"一填就对、一试就懂"，是 Excel 式护城河的根；时序分析的窗口长度、趋势判定阈值本身需专业判断、易配错、试算不直观。**不让高复杂度能力拖累引擎"简单可用"的本性。**

> 补丁5是首个明确分期的补丁：留位置（架构）≠ 全实做（工程）。

### 【补丁6】规则时间有效性（as-of 计算）—— 金融验证逼出

现有 schema 只有 `meta.version`（整包一个版本号）。金融打破它：监管要求"用授信当时那版规则重算"——需要单条规则带生效期、引擎能按历史时点选规则。**这是地基缺口，非呈现问题**：行业模板能让界面露出日期字段，但"穿越回某时点选当时有效规则计算"是引擎逻辑，呈现层变不出来。

- rule 增加 `effective_from` / `effective_to`（单条规则的生效/失效日期）。
- 引擎支持 `as_of` 计算时点：默认当前；可指定历史时点，自动选当时有效的规则集。
- 结果记录"本次采用的规则版本"，接回"证据+确认"贯穿线与守护。

**分期落地**：schema 现在留好 effective_from/to 与 as_of 位置（架构对）；第一版引擎实做"当前时点 + 记录所用版本"，**把"任意历史时点回溯重算"标为 v2**（回溯需存历史数据快照，工程重，不拖累第一版）。

> 适用：金融/保险/税务/医保结算 等所有"按当时规则追溯审计"的强监管行业。

## 第三层 formulas（公式）

```
Formula {
  id          // 如 F_风险指数
  label
  type        // weighted_sum 加权和 | normalize 归一化 | product 乘积 | piecewise 分段映射
  expression  // 受限表达式：只能引用已定义的 metric / formula；只能用白名单运算
  output_kind // number（多数公式产出数值）
}
```

**受限表达式约束**：只能引用已定义的 metric/formula id；只能用 + - * / 和白名单函数（加权、归一化、分段）；**禁止任意脚本**。配错最多算错数，不会崩、不会注入。

## 第四层 rules（触发规则）

```
Rule {
  id            // 如 R-LEVEL-D-001
  label
  when          // 条件表达式：引用 metric/formula，用比较符 + AND/OR 组合
  then          // 【补丁1】结论，支持多类型（见下）
  priority       // 优先级数值，越大越先裁决
  is_veto        // 【补丁3】是否一票否决（命中即定，先于打分）
  effective_from // 【补丁6】生效日期（as-of 计算用，可空=一直有效）
  effective_to   // 【补丁6】失效日期（可空）
  evidence_ref   // 触发依据（证据ID列表）
}
```

### 【补丁1】then 结论支持多类型 conclusion_type

跨行业验证发现结论不止"分级"一种，引擎必须支持三类：

| conclusion_type | 含义 | 行业例 |
|---|---|---|
| `grade` | 分级/贴标签 | 药监 风险等级=D |
| `decision` | 决策 | 贷款 审批结果=拒绝/通过 |
| `assign` | 赋值（算出具体值） | 贷款 额度=月收入×8倍；核保 保费系数=1.3 |

一条规则的 then 可以同时给出多个结论（如"通过 + 额度=X"）。

---

## 【补丁2】conflict_policy（冲突裁决策略）—— 一等公民

多条规则同时命中时怎么裁决。**每个行业默认取向不同，必须可配。**

| 策略 | 含义 | 适用行业例 |
|---|---|---|
| `most_severe` | 取最严结论 | 药监（取最高风险） |
| `veto_first` | 一票否决优先（任何 is_veto 命中即采用） | 贷款（任一拒绝规则命中即拒） |
| `by_priority` | 按 priority 数值高者 | 通用 |
| `weighted` | 加权综合 | 评分型场景 |

---

## 贯穿线 evidence_policy（证据与人工确认）—— 引擎强制，非行业可配

**这一段药监和贷款一字不差，证明它是真通用，所以由引擎强制、不开放给行业改。**

```
evidence_policy: {
  require_evidence_ref: true,       // 每个结论必须回指证据，否则不可成立
  default_status: "待人工确认",      // 所有结论默认待确认，人确认后才生效
  completeness_metric: true,        // 计算"证据完整度 = 已绑定/应绑定"
  no_auto_conclude_when_incomplete: true  // 关键证据缺失的事项，标"待核实"，绝不自动下结论
}
```

## 执行流程（引擎运行时按此跑，跨行业不变）

```
1. 加载 RulePack（若 meta.enabled=false，本项目无此步，纯CRUD）
2. 按 data_bindings 取该对象的真实数据
3. 算 metrics（含证据回指）
4. 算 formulas
5. 评 rules，得出所有命中结论
6. 按 conflict_policy 裁决出最终结论
7. 套 evidence_policy：绑证据、算完整度、缺证据项标"待核实"、整体置"待人工确认"
8. 输出结论 + 证据链 + 待确认状态
```

## 变更记录（来自跨行业验证的修正）

| 补丁 | 内容 | 被哪个行业逼出来 |
|---|---|---|
| 补丁1 | 结论支持多类型（grade/decision/assign），不止分级 | 贷款的"额度"塞不进纯分级 |
| 补丁2 | conflict_policy 升为一等公民，可配裁决策略 | 贷款"一票否决" vs 药监"取最严" |
| 补丁3 | rule 增加 is_veto 字段，支持一票否决 | 贷款的拒绝规则需先于打分 |
| 补丁4 | metric 增加 source_type，支持人工录入的定性指标 | 医疗的专家现场打分非数据算出 |
| 补丁5 | metric 增加 metric_family(aggregate/temporal)，支持趋势类指标；并首次分期(留位置≠全实做) | 电网的"值未超标但持续恶化"预警 |
| 补丁6 | rule 增加 effective_from/to + 引擎 as_of 时点计算，支持按历史规则版本追溯；分期(回溯重算标v2) | 金融"用授信当时那版规则重算" |
| —— | 安全生产：**0 架构补丁，全程复用**。二维风险矩阵被剥为 product+piecewise 两步表达，不规则矩阵的 matrix_lookup 留作 v2 可选原语(留位置不实做) | 信号：地基收敛 |

> 这些若只看单一行业样例都会漏掉。**串行验证**（每个行业站在前一个修正过的地基上）的价值正在于此：
> - 贷款逼出补丁1/2/3；这三个补丁在医疗的"高分被一票否决"场景里**直接救场**，所以医疗没在否决逻辑上逼出新东西。
> - 医疗在"否决/嵌套"上扛住了（前序补丁的复利），却在"指标来源"上逼出了补丁4。
> - 已验证行业：药监(原型) → 贷款(补1/2/3) → 医疗(补4) → 电网(补5) → 金融(补6) → 安全生产(0补丁,收敛)。
> - **收敛判据**：补丁数随行业递减 3→1→1→1→0。安全生产全程复用、零架构新增 = 地基收敛信号。政务(事件流聚合)预判落在现有 aggregate 族，继续磨边际收益≈0。
> - **结论：schema v1.3 在此封版**，转入引擎与配置界面工程。后续行业靠"已有能力组合 + 行业模板配置"消化；只有再出现连组合都表达不了的实质新维度，才解封改地基。
> - **另立机制：行业模板(Industry Profile)**——选行业=加载该行业预置模板(预设可用原语+预填指标/公式/规则骨架+收起用不到的高级选项)。管"呈现层(每次露多少)"；schema 管"地基层(能力全不全)"。两层各管一段、不可互相替代(金融案例已证)。
> - **判断缺口真假的原则**：剥到数学/逻辑实质再判，别被业务表象吓住(安全生产二维矩阵=product+piecewise，无需新原语)。否则地基为每个"看起来不同"膨胀成谁也配不动的怪物。
