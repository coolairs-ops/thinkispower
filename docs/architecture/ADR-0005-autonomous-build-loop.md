# ADR-0005: 自治建造回路（把 coding-agent 的 SOP 变成平台建造引擎）

**Status:** Accepted（方向规格 / SOP 引擎北极星；逐步落地，先 MVP）
**Date:** 2026-06-19
**Deciders:** 平台负责人 / 架构 owner
**关联:** 承接 [ADR-0002] 原则①（固定归确定性、LLM 只补）与柱五（智能编排）；复用 [ADR-0001 路 B] 与分段生成（commit f7a7ee1）。是"生成中等复杂度程序"这一顾虑的系统解。

---

## Context（核心顾虑与受力）

平台负责人最大的顾虑：**设计思想动力时能逐环节详尽细致；但无法保证思想动力生成"其他产品"时，每个环节也这么细致。** 即——**没法让每次 LLM 调用都像人一样想全每个环节。**

本会话的实证（为什么单靠"控制论自迭代"不够）：
- 自迭代是"单产物抛光循环"，传感器量的是 HTML 结构/场景通过率（代理信号），autoFix 是有损执行器 → **实测把 demo 越改越差（健康分 84→81），加防退化护栏才止住**。
- 控制论能优化**可测误差信号**，但"设计完备/想全环节"不是误差信号，**造不出来**。传感器/执行器一弱，闭环就"越控越坏"。

**关键洞见（反转）：严谨在"回路"，不在"单次调用"。** 你无法让每次 LLM 调用都细致；但你能让**回路**像你一样有纪律，用回路逼每个有界步骤过门。这正是 coding agent 在确定性 harness 下的工作方式：**harness 持有循环，LLM 只做有界一步，状态全外置、跨上下文窗口续跑**。严谨来自"被设计一次的循环"，循环把纪律强加给每个被生成的产品。

---

## Decision

把平台的生成引擎，从"一次性生成 + 抛光循环"，改造成 **确定性建造编排器驱动的「逐模块自治建造回路」**：

- **回路确定性、LLM 只做有界一步、状态全外置**（ADR-0002 原则①在"建造过程"层的落地）。
- 编排器（确定性代码）遍历模块 DAG，对每个模块跑：`生成 → 测试门 → 视觉门 → 写交接日志 → 人审门(按风险) → 标记 done → 下一个`。
- **状态外置到库**（建造计划 + 交接日志 + 产物），使建造**跨上下文/会话边界续跑**——LLM 永不需 hold 整个建造。

> 一句话：**把"我（设计者/coding agent）的工作 SOP"编码进确定性回路，让平台对每个产品自动执行这套纪律。**

---

## SOP（coding-agent 工作循环，平台据此实现）

1. **先读后改**：动手前读真实代码/状态，不假设。
2. **切成有界增量**：大活拆成独立可验证的小片。
3. **外科手术式改**：只碰必要处，顺现有风格。
4. **每片立刻验证**：跑测试 + 构建 + 看行为；**没绿不算 done**。
5. **状态持续外置**：写记忆/交接/commit——状态在库/文件，不在上下文里。
6. **自然边界 checkpoint**：逻辑单元留存、给小结。
7. **不可逆/含糊处人在回路**：摆权衡、求确认，不偷偷决定。
8. **诚实报告**：过没过带证据，明说延后，不假装 done。
9. **从反馈自纠**：发现负优化→查根因→加护栏→重验。

## SOP → 平台机制映射（现状 / 待建）

🟢已有 🟡部分 🔴待建

| SOP 步 | 平台机制 | 状态 |

| 4 **视觉门** | 渲染→截图→vision 或人判（screenshot-replicate/vision 已有） | 🟡 |

| 6 继续下一个 | 编排器读日志+任务态，从 DAG 挑下一 ready 模块 | 🔴 |

## 外置状态的数据模型（让它能续跑的关键）

```
BuildPlan      模块 DAG
  modules[]: { id, name, spec, deps[], status }
  status: pending → building → testing → visual → review → done | blocked
  topoOrder

BuildJournal   append-only 交接日志（= ADR/记忆/小结的产品内化）
  每条: { moduleId, phase, summary, decisions, knownIssues, testResult, shotRef, ts }
```

编排器**无状态地**读 BuildPlan + BuildJournal 决定下一步 → **库是真相源，LLM 无需记忆**。这就是"上下文满了换窗口还能接着干"的本质：状态不在任何单个 LLM 上下文里。

---

## 控制门与诚实的天花板

三道门让"done"是真的：
- **测试门**：CRUD/UI 用传感器 + 场景；深业务逻辑必须**让 LLM 连测试一起生成**，否则门是假的。
- **视觉门**：截图 + vision/人。
- **人审门**：模块边界，按风险分级。

> **天花板 = 每个模块"门"有多真。** 路 B 那种（数据+CRUD+UI）门够强 → 中等复杂度站得住；深业务逻辑门会变虚 → 人审补、且必须生成真测试。**控制论不是魔法，只是把"门"自动化；门真→站得住，门虚→它会假装合格（=本会话的负优化）。**

---

## Consequences

**变容易：** 严谨锁进确定性回路而非指望 LLM；建造跨上下文/会话续跑→可啃中等复杂度；每模块带门、可追溯、可人审→对"可信交付"是结构性加强。
**需关注：** 门的真实性是上限（测试门弱=假合格）；编排器+日志+续跑是实打实工程量；模块 DAG 的依赖/拓扑要正确，否则乱序生成。
**将来重访：** "LLM 自动生成模块测试"的可靠性；视觉门的 vision 判别准确率；blocked 模块的人工接管 UX。

---

## MVP 最小可交付（基于现有件）

**做：**
1. **BuildPlan + BuildJournal**（库）：扩 DecisionLog/autoIterateState；模块 DAG + append-only 日志。
2. **确定性编排器 v1**：遍历 DAG → 每模块 `分段生成 → 测试门(传感器/场景) → 写日志 → 下一个`，**从日志续跑**（先把"测试门 + blocked + 续跑"做实）。
3. **测试门做真起步**：CRUD/UI 模块用传感器 + 验收场景判定 done/blocked。

**不做（显式延后）：** 视觉门自动化、深业务逻辑的自动测试生成、模块边界全自动人审 UX、并行建造。
