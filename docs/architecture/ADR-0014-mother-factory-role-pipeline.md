# ADR-0014: 母体角色流水线 —— 判断型提示词化 / 机械型确定性化 / 接缝设门，确定性优先

**Status:** Proposed（2026-06-25；把"从需求到上线的隐性角色"显式化为可治理流水线，按确定性优先拆分。**实现时机排在客户系统脊柱跑通 + 首批真交付之后**，本 ADR 先把施工图定下来）
**Date:** 2026-06-25
**Deciders:** 平台负责人

**关联:** 是工程控制论闭环的"完成式"——把控制器/执行器/传感器/反馈四角色细化为命名工位；承 [ADR-0009] 确定性门、[ADR-0012]/[ADR-0013] 确定性优先纪律、[ADR-0005] 自治建造回路、[ADR-0008] 能力复利（角色库同构于词汇生长）；用 [skill-creator/eval] 作判断型角色的版本化与评测载体。

---

## Context（为什么要显式化角色）

当前"从需求到上线"的中间角色——业务分析、架构、codegen 操作、QA、运维守护——**大量隐式地压在"大模型通用能力"上**：一段大 prompt 啥都干，质量看模型当天发挥，不可复利、不可单测、接缝处无门。

**病灶一句话**：角色隐式 + 未区分"判断 vs 机械" + 接缝无门 → 质量看天、错误层层放大到交付才暴露、每个项目从零重来不沉淀。这与本平台"靠门不靠提示词口头护栏""确定性优先"的纪律相悖。

**关键判断**：把链路显式拆成命名角色，**不等于全用大模型**。恰恰相反——拆分的第一价值是**把该确定性的角色从大模型手里夺回来**（codegen 操作、上线门、脱敏等本就该是代码/门），只把真需要判断的少数角色提示词化。这是"确定性优先"在角色层的应用。

---

## Decision

### D1｜角色清单（4 段约 18–20 工位，每个标 J=判断型 / M=机械型 / H=混合）

**需求段（判断密集，提示词化收益最大）**
- 需求访谈师/业务分析师：口语→结构化 IR 【J】
- 完备性审查官：挑缺的页面/实体/字段/权限，每条带证据 【J】
- 缺口处置决策者：每缺口判 autofill/ask/info 【J】
- 领域建模师/数据架构师：IR→dataModel + 关系（主子表/树/N-N）【J】
- 权限架构师：角色/数据范围矩阵（当前最弱环，承 ADR-0012 D2）【J】
- 方案/产品经理：页面清单/PRD/方案确认 【J】
- 交付形态路由师：一个界面 vs 两个（deliveryProfile，ADR-0013 口二）【H：规则为主 + 判断兜底】

**构建段（机械为主，禁止提示词化）**
- codegen 操作员：importTable→写工程→编译→重启 【M·确定性】
- 装配工程师：菜单/RBAC seed、契约对齐 【M】
- 后端开发（仅超 CRUD 时）：领域 AI 端点 【J，ADR-0011 未建】

**质量段（已做对——是门，不是提示词）**
- 编译门(L1)/运行时测试(L2,冒烟·backend-smoke)/语义验收(L3)/契约审查官/上线门决策官 【全 M·确定性门，上线门见 ADR-0009】

**运维段**
- 守护巡检员（定时体检）【M】/ 故障分级决策者（四级 classify）【M·规则】/ 修复执行回滚员【M + 适配器】/ 健康报告撰写员【J】

**贯穿**
- 编排官（hermes：任务 DAG/状态外置/续跑）【H】
- 脱敏/对外话术官（规则表）【M】

> 判断型（J）约 7–8 个、几乎全在需求段；机械型（M）十余个。**只提示词化 J，硬编码 M。**

### D2｜拆法纪律（确定性优先，核心）
1. **判断型 → 提示词化**：每个 J 角色 = 聚焦提示词 + 输出契约(schema) + few-shot 正反例 + 自带校验。
2. **机械型 → 确定性代码/门**：M 角色绝不提示词化（codegen=importTable、上线门=二值闸、脱敏=规则表）。**把 M 塞回提示词 = 把刚靠若依/确定性门赶走的不确定性请回来——红线。**
3. **每个接缝设门**：相邻角色之间插一道校验（QA 段已如此，向需求段延伸）。错误早抓，不层层放大。接缝交接走干净契约（AppSpec/DataContract，依赖 ADR-0013 口一的中立化）。

### D3｜不过度拆分（产能是粒度的 U 型曲线）
角色过多 → 串行延迟叠加、错误沿链传播、协调开销暴涨，**净亏**。粒度取"一个角色 = 一个可独立验收的判断/动作"，不再细分。先粗后细，按证据加细。

### D4｜复利：判断型角色 = 可版本/可评测的角色库
J 角色提示词进版本库，用 eval harness 单独打分、A/B、回归（skill-creator/eval 正为此）。**改一次角色提示词，所有未来项目复利**（同构 ADR-0008 词汇生长）。这是"母工厂"杠杆的实处。

### D5｜按角色路由模型（经济杠杆）
M/抽取型角色用便宜模型、判断重的 J 角色（架构师/审查官）用强模型。**成本与延迟一起降**——从"每项目烧一个贵会话"变成"流水线上贵的只占几站"。

### D6｜时机（见下节"何时做"）
本 ADR（施工图/角色清单/标注/接缝门）**现在即可落档**，零成本、指导后续。**实现（逐角色提示词化 + eval + 接缝门）排在客户系统脊柱跑通 + 首批真交付之后**，增量逐角色推进，不与脊柱抢稳定性。

---

## 何时做（时机，重要）

**不现在做实现，原因是硬依赖而非偏好：**
- **拆分要对着证过的链路**：哪些角色真承重、哪些接缝真漏、哪些已是确定性、哪些是飘忽 LLM——**只有把整条链端到端真跑一次（= 客户系统脊柱 Phase 0–4）才看得清**。脊柱未通就拆分，等于对着未验证的流水线下刀，与"先修空跑再升门"的纪律相悖（同守护执行器、ADR-0010 的教训）。
- **正确顺序**：① 脊柱跑通（一个真上线的客户系统）→ ② 跑 1–2 个真实交付，记录质量/产能真实瓶颈在哪个角色 → ③ 按证据**从需求段判断型角色起**，一次提示词化 + eval 一个角色，接缝插门，灰度合入。
- **唯一现在该做的**：把本 ADR 的角色清单 + J/M 标注 + 接缝门**写下来**（文档，零成本），作为后续施工图与 Claude Code 改造依据。

**先做哪几个角色（实现期排序建议）**：权限架构师（最弱、ADR-0012 D4 验收要它）→ 领域建模师 → 完备性审查官/处置决策者。这几个是需求段判断密集、当前最依赖大模型通用能力、复利最高的。

## 执行手册（到点照做 · SOP）

> 目的：把"建角色"这件事也确定性化——下面是可照做的步骤、模板和门槛；**执行时无需重新设计流程，只填内容**。哪些是"照做"、哪些"必须人拍板"，分清在 R3。

### R1｜角色定义模板（每个角色注册前必填，机器可校验）
```yaml
role: 权限架构师            # 角色名
stage: 需求段               # 需求/构建/质量/运维/贯穿
type: J                     # J=判断(提示词化) / M=机械(确定性代码) / H=混合
consumes: 结构化IR + dataModel      # 输入契约（引用 AppSpec/DataContract 字段，不自定义私有结构）
produces: 角色/数据范围矩阵(schema)  # 输出契约（必须是 schema 化结构，非自由文本）
gate: 契约校验 + 矩阵完备性门         # 本角色产物后接的那道确定性门
model_tier: strong          # strong / cheap（按 D5 路由）
eval_suite: evals/role-perm-architect/   # 该角色的 eval 用例目录
status: draft               # draft / eval-passing / production
```
> M 型角色只填到 `gate` + 指向其确定性实现文件，**不写提示词、无 eval_suite**。

### R2｜建一个 J 角色的标准流程（照此 9 步，每个判断型角色都一样）
1. 填 R1 模板（定义先于实现）。
2. 定**输出 schema**（结构化，不许自由文本）——这是后接门能校验的前提。
3. `skill-creator` **脚手架**出提示词初稿（AI 起草，**仅初稿，不算数**）。
4. **写 eval 用例**：从真交付取黄金输入 → 人工标注期望输出（这步必须人，见 R3）。
5. 跑 eval harness 打分。
6. 未过阈值 → 改提示词 → 回 5（只改提示词，不动 schema/门）。
7. 过阈值（R4 判据）→ 人工签字（R3）→ 注册进角色库，`status=production`。
8. 在该角色与下游之间**接上 R1 声明的门**（接缝校验）。
9. 配 `model_tier`，记录该角色 cost/延迟基线。
> 任一步不过 → 该角色停在 `draft/eval-passing`，**绝不裸用未过 eval 的提示词上生产**（同上线门纪律：算数靠门不靠初稿）。

### R3｜必须人拍板的点（写死会错，执行时由人填/定）
- **瓶颈定位**：先做哪个角色，依据真交付证据（Action Item 2 的量化），不预设。
- **黄金 eval 用例**：什么算"正确输出"要人标注（造用例的*流程*已模板化，*内容*靠人）。
- **阈值设定与微调**：每个角色 eval 通过线由人定，按返工率回调。
- **提升生产的签字**：`eval-passing → production` 需人确认（防"分高但实际不对"）。

### R4｜eval 门通过判据（默认值，可按角色覆盖）
- 结构合规率 = 100%（输出必须过 schema，硬门）。
- 黄金集语义正确率 ≥ 阈值（默认 90%，与平台 ≥90 评分纪律对齐）。
- 回归不劣化：改提示词后旧黄金集不得跌破上次分。
- 真交付返工率（该角色产物被下游门驳回比例）≤ 设定值。

### R5｜角色库落位（与 ADR 分离）
- 角色**定义**（R1 模板）：随本 ADR / 一份 `role-registry` 文档，稳定、评审可见。
- 角色**提示词 + eval 用例**：独立版本库（一个 J 角色 = 一个 skill 目录 `roles/<name>/`），可单独 eval/A-B/回归/模型路由。
- **ADR 不存提示词**；提示词改动走 eval 回归，禁止裸改。

---

## 边界（管什么、不管什么）

- **管**：把链路角色显式化 + 按"判断/机械"分类 + 定拆法纪律（提示词化判断、硬编码机械、接缝设门）+ 复利/模型路由策略 + 实现时机 + **建角色的可照做 SOP（R1–R5）**。
- **不管**：本轮不实现任何角色的提示词化/编排框架（等脊柱通 + 真交付证据）；不引入重型多智能体编排中间件（YAGNI，先用现有 hermes 编排）；不改 M 角色现有确定性实现。

---

## Consequences

**产出（质量）上升**：聚焦角色 + 接缝门 + 复利改进，需求段判断质量台阶式提升（当前最弱处）。**产能上升（有条件）**：并行 + 按角色路由模型 + 缓存复用——但**仅在不过度拆分时**成立。**可信/可治理**：命名角色 + 接缝门 + 留痕，直接兑现"可信交付/可审计"叙事，把控制论工位从隐式变显式。**复利**：每改一个角色，全量未来项目受益。

**代价/风险**：角色库 = N 份提示词要养/评测/版本（规模化才回本）；接缝契约不干净则拆分反伤质量（依赖 ADR-0013 口一）；过度拆分反噬产能（D3 守住）；提示词化机械角色会回灌不确定性（D2 红线）。

## Action Items

1. [ ] **（现在·零成本）** 本 ADR 角色清单 + J/M/H 标注 + 接缝门 + **执行手册 R1–R5（模板/SOP/拍板点/门判据/库落位）** 定稿；作为母体角色流水线施工图。
2. [ ] **（脊柱通后·先决·R3 瓶颈定位）** 跑 1–2 个真实交付，按角色记录质量/产能真实瓶颈（量化：projects/day、各门通过率、各角色返工率、cost/project）。
3. [ ] **（实现·逐角色·按 R2 九步）** 从权限架构师起，逐个 J 角色：填 R1 模板 → 脚手架 → 写 eval → 过 R4 判据 → 接缝插门 → 灰度合入。
4. [ ] **（经济）** 给每个角色配模型档（M/抽取→便宜、J 重→强），度量 cost/project 降幅。
5. [ ] **（复利）** J 角色提示词进版本库，纳入 eval 回归，禁止裸改不评测。
6. [ ] **（红线·防过度建设）** 明确记：M 角色不提示词化；不引重型多智能体框架；不在脊柱通前动实现。
