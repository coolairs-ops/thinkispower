# ADR-0006: 思想动力母体化 —— 配件仓库 + 行业包 + MCP 能力网关 + agent 分层

**Status:** Accepted（政企方向总纲；2026-06-20 经多轮研讨收敛拍板）
**Date:** 2026-06-20
**Deciders:** 平台负责人

> **实施进展（2026-06-20）**：装配线地基已实测打通——`provisionApp`(实体→真若依后端)、适配器①(IR→AppSpec，现有项目一键嫁接)、适配器②(前端 serve 时按 backendRuntime 自动切若依真数据)、"用若依做真后端"流程入口已落、全仓测绿。**已形式化为 [ADR-0007] 应用契约先行（Accepted，2026-06-21）**：把"填槽+校验门"纪律从后端 codegen 扩到**前端迭代**——应用契约统一驱动前端生成+后端置备+验收门，让迭代朝若依契约收敛（现迭代对若依是盲的）。写入翻译已落（commit d24c975），并经 ② 若依全 API 写实测 live 证通（4 资源最小创建全 200）。
**关联:** 落实 [ADR-0002] 柱二（行业规则包）+ 柱四（外部适配器）+ 柱五（智能编排），给这两根此前 🔴 空着的柱子**具体形状**；建在 [ADR-0003] 若依强制底座 + [ADR-0005] 建造回路之上；消费 ADR-0003 M3c 刚打通的**全自动 provision**（建表→codegen→部署→重启→RBAC，实体级实测通）。

## Context（为什么现在定这条）

- **政企命门**：政企/监管类系统的价值密度在**行业规则 + 外部数据接口**里（药监风险画像实证：80% 价值在规则包/证据模型/数据接入，外壳只占 ~20%）。这两块是 ADR-0002 标 🔴 的两根空柱子，不解决就只能做"漂亮外壳、接不进数据、扛不住执法"的半成品。
- **关键洞见（裁定）**：这些能力（规则引擎、抽取引擎、知识库、地图/语音/图片/视频…）**是"生成的产品"要具备的能力，不是母体自身的中枢能力**。母体是**资源池 + 装配工厂**，把它们当**配件**按需接进产品。私有化（数据不出域、可气隙）要求产品**自包含、不回连母体**——"配件装进产品"天然满足，"产品依赖母体中枢"则违背。
- **优化裁定（本 ADR 的更优版本，非首版）**：经多角度论证，否决"自建三引擎 + 全 codegen + 全 MCP"的重方案，采用"**集成优先 / 领域层配置驱动 / 内容即护城河 / 双轨 MVP / MCP 兼容不锁死 / 确定性优先按需 agent**"的轻方案。

---

## Decision（六条焊死）

### D1｜母体三层模型 + 配件两类用法
- **三层**：**母体（资源池 + 装配工厂）→ 配件仓库（标准件）→ 产品（自包含装配体）**。母体按需求**选配件 + 接入 + 生成**产品；产品在客户环境独立跑。
- **配件两类用法**（不许混）：
  - **装进产品**（产品运行期需要）：规则引擎实例、抽取引擎、知识库、能力连接器——**随产品交付、跑在客户侧**。
  - **母体生成期自用**：Hermes 理解/拆解/验收、需求补全 A/D/E、建造回路——**留在工厂，不进产品**。
  - 有些配件两边都用（同一抽取引擎：母体建包时用、产品运行时也装一份）。

### D2｜集成优先于自建（配件能买/组装就不造）
- 规则引擎/RAG 知识库/工作流等是 commodity，有成熟政企件（规则引擎候选：**URule**（中文、可视化规则设计器、政企友好）/ LiteFlow / Drools；RAG/向量库用成熟本地方案；工作流用若依自带）。
- **母体只自建别人替不了的三样**：① 配件**接口契约** ② **装配线**（行业包→选配→接入→provision）③ **出域/审计治理**。**禁止重造规则引擎/向量库等轮子。**

### D3｜领域逻辑层配置驱动，仅实体骨架走 codegen（混合管线再进一层）
- 承接 ADR-0003 混合管线，**再推进一层**：
  - **实体骨架 / CRUD / 权限**（低频变）→ **codegen（若依路 C）**。
  - **规则 / 评分 / 画像 / 证据 / 动作**（高频变）→ **运行时元数据驱动（路 B 哲学）**，改包**不重编译**（规则引擎本就是规则即数据）。
- 收益：行业包成为**可热调的活资产**（改规则/调阈值/换模板秒级生效，免 ~11min rebuild），"持续生成报告"才真正持续。

### D4｜能力网关 = MCP 兼容契约 + 出域治理层 + 审计层
- **地图/语音/图片/视频/OCR/其他 API = 可插拔工具（能力插件）**，与"外部数据连接器（进数）"分属两种抽象。
- **能力契约设计成 MCP 兼容**（能力做成 MCP server 即插，引擎零改），**但保留直连抽象、不硬依赖 MCP 运行时**（"MCP-compatible, not MCP-locked"，对冲年轻标准风险 + 关键能力可直调省开销）。
- **必须自包两层 MCP 不给的**：① **出域治理**（政务默认禁外部 server、本地优先、出域先脱敏）② **审计**（每次能力调用留痕，供执法审计）。
- 能力网关服务于 **agent 层调工具**；**确定性规则引擎不走 MCP**。

### D5｜agent 分层，确定性优先、按需加 agency
- **规则引擎 = 确定性，不是 agent**（执法判断必须可解释/可审计/可复现；刻意不让 AI 决定风险等级/动作）。
- **抽取引擎 = 有界（v1 先做纯确定性 schema 抽取，不上 loop；agency 按需、按证据再加）**，强约束：schema + 来源定位强制 + 人在回路确认。
- **编排层（理解/分解）= 受控 agent**，复用 ADR-0005 成法（确定性编排器 + 测试门 + 状态外置 + 人确认），**不放飞**（自迭代放飞实证负优化 84→81）。
- 铁律：**严谨在回路（门/校验/人确认），不在单次 LLM 调用；能确定性绝不用 agent。**

### D6｜资本配置：押行业包内容 + 可信运营，不押引擎
- 引擎/网关是 commodity、有替代、非护城河。真护城河 = **① 行业包内容资产**（药监规则本体/证据模型/法规知识库/画像模板，做一次卖 N 省局，边际成本趋零）**② 可信交付 + 持续守护的运营壁垒**（承接商业判断"值钱的是可信交付+持续守护"）。
- **资源压在"行业包内容生产方法论 + 可信运营"，不压在造引擎。**

## 行业包规格（声明式 bundle，插件 / 预置同一格式）

```
IndustryPack {
  meta:        { id, name, domain, version }
  dataModel:   AppSpec(entities + relations + roles + menus)   // 复用现有 IR + provisionApp（已通）
  rules:       RuleDef[]          // 触发规则 + 评分公式 → 喂规则引擎配件（URule 等）
  extraction:  ExtractionRecipe[] // 文档类型 → schema → prompt + 来源定位 → 喂抽取配件
  templates:   PageTemplate[]     // 画像/报告页 → 复用 demo 生成/截图复刻（已有）
  actions:     ActionDef[]        // 动作目录 + 触发条件 → 规则引擎/动作台账
  workflow:    WorkflowDef        // 确认流 → 若依工作流（已有）
  capabilities:CapabilityRef[]    // 依赖哪些能力插件（地图/语音/视觉…，经能力网关）
  connectors:  ConnectorRef[]     // 依赖哪些外部数据连接器（进数）
  knowledge:   KnowledgeRef[]     // 知识库配件（RAG 数据包）
}
```
- **插件 vs 预置 = 同一份包的两种分发**：私有化交付把包**预置**进镜像；做生态就**插件**化加载。**先定包规格，两者自然都成立。**
- provision 扩一步：按 `pack.capabilities/connectors/knowledge` 声明，**装配线把对应配件模块纳入产品构建**（若依 starter 模块 / 微服务 / 数据包）。

## 对 ADR-0002 的修订
- **柱二「行业规则包」**：由 🔴 概念 → 给出**包规格 + 配件化 + 集成优先**的具体形状；首个包 = 药监（同时是建包方法论的样板）。
- **柱四「外部适配器」**：拆成**两种抽象**——「外部数据连接器（进数）」+「能力插件（处理，经 MCP 兼容网关）」。
- **柱五「智能编排」**：明确为**受控 agent**（ADR-0005 成法），非放飞。

## 落地次序（双轨 MVP，锁方向不砸现有闭环）

1. **轨 A（外壳优先，最快出样板）**：定**包规格** → 药监包的 dataModel+模板+权限+工作流**全用现成能力** → 经 `provisionApp` 出"**包 → 可演示画像系统**"（数据先样例）。**复用已通的全自动 provision，零额外引擎即可出第一个样板。**
2. **轨 B（硬骨头并行，真命门）**：**数据接入（连接器框架）+ 执法级证据可信**。监管类胜负手在"数据进得来、证据扛得住"，不能等轨 A。
3. 接**规则引擎配件**（集成 URule 等）→ 画像有真分数真等级。
4. 接**抽取引擎**（确定性 schema 抽取先行）→ 文档→证据→人工确认入池。
5. **能力网关（MCP 兼容 + 出域/审计）** + 地图/语音/视觉按需接。

> 触发条件：轨 A/规则引擎随首个行业包（药监）真单子拉动；轨 B 连接器随具体省局对接拉动。承接 ADR-0002 原则④（重集成等真实单子兜底），但**包规格 + 装配线属地基，可先立**。

## Consequences

**正面**
- 两根空柱子有了具体、可落地形状；政企方向有总纲。
- 集成优先 → 砍掉 ~60% 自建工作量与最大技术风险。
- 领域层配置驱动 → 消除 rebuild 痛点，包成活资产。
- 母体/配件/产品分层 + 私有化自包含 → 数据不出域天然满足。
- 内容即护城河 → 资本配置对，可复制到 N 省局/N 行业。

**代价 / 风险（诚实）**
- 配件要真能独立打包 + 带运行时依赖（私有化部署）+ 版本契约——每配件一次性工程。
- 连接器对接政府系统首次仍是现场集成活（框架降边际成本，不消除首次）。
- 行业包内容要专家共建（首包真投入；框架让其可复用）。
- 执法级证据可信是更高标准，要专门打深。
- MCP 年轻——靠"兼容不锁死"对冲。
