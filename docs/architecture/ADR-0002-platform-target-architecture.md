# ADR-0002: 平台目标架构（底座 / 规则包 / 三层生成 / 适配器 / 编排）

**Status:** Accepted（作为**北极星目标架构 / 方向锚点**；不是"立即全建"的承诺——逐柱**按真实需求增量建**）
**Date:** 2026-06-19
**Deciders:** 平台负责人 / 架构 owner
**关联:** 建立在 [ADR-0001 固定后端运行时（路 B）] 的脊柱上；统一"路 A/B/C"与"Phase 0–4 路线图"的目标态；与 `docs/TRUSTWORTHY_CREATION_ROADMAP.md` 互为里子（路线图给阶段，本 ADR 给目标架构与触发条件）。

---

## Context（为什么要这份锚点）

平台演进到现在，已有一份完整的"顶层架构交底"（智能体时代的应用开发操作系统：底座/规则包/三层生成/适配器/编排五柱）。它逻辑自洽、且与日常对话收敛的结论同源。但它是**未来态蓝图**，若不落成锚点会有两个风险：

1. **跑偏 / 过度工程**：把 2–3 年北极星当成"现在该一次性建"，违反"先稳定再扩张、按真实需求增量"的戒律。
2. **重复造**：已建成的（守护/可信引擎/路 B/双模式雏形）被当成待办重做。

本 ADR 把目标架构钉成锚点：**每根支柱标清「现状 / 未来态 / 什么时候建」**，让每一步对照它判断"是朝它走，还是偏了/重了"。

**一句话目标隐喻**：不是"一个能生成软件的 AI 工具"，而是"一块能让软件在其上安全生长的土壤"（类比 Android：不生产 App，提供框架/服务/规范）。
**职责分工**：**底座管、规则包限、适配器连、编排想、生成做。**

---

## Decision（决策）

采纳"五柱 + 四层"作为**目标架构**，并确立四条**贯穿性原则**（同时是对原交底的修正）：

- **原则①｜固定的归确定性，不确定的才归 LLM。** 底座、CRUD、行业硬规则、外部对接，一律用**确定性手段**（固定运行时 / 代码生成器 / 校验器 / 适配器）承载；LLM **只补业务差异层**。生成范围越窄，故障面越小。
- **原则②｜hard enforcement 靠校验器/codegen，不靠提示词。** 行业"强制规则"**不能**指望 LLM 照做（已实证：LLM 连更丰富的提示词都不充分照做，受 8K 输出上限 + 合规度所限）。强制 = **确定性 codegen 产出 + 传感器/门控驳回不合规产物**。
- **原则③｜契约即普通代码，后端可替换（承接 ADR-0001 两条硬约束）。** 数据模型用真 Prisma、对外用真 REST；脊柱按 `BackendRuntime` / 适配器接口编程，换实现不换契约。
- **原则④｜北极星增量建，由真实需求拉动。** 任何重投入（若依/路 C、规则包引擎、适配器框架、私有化包）**等真实试点（以岭）拉动且单子兜底**才启动，不投机预建。

## 五柱：现状 / 未来态 / 什么时候建

> 现状均为**代码实证**（截至 2026-06-19）。🟢已建 🟡部分 🔴未建

### 柱一｜统一底座（强制骨架）
- **未来态**：人工建一次的成熟内核——组织/RBAC 权限/行级隔离、不可变审计日志、审批流引擎、通知中心、文件存储、定时任务、数据字典、动态菜单。**生成的应用禁止自建权限/日志**，必须复用。
- **现状 🟡**：路 B `app-runtime`（NestJS 通用 CRUD，元数据驱动，per-project schema 隔离）+ 多租户雏形（org/membership/tenant-scope）。**缺**：per-app 鉴权、审批流、通知、强制审计、字典、动态菜单——即"薄底座，非强制内核"。
- **什么时候建**：成熟内核 = **若依-Plus**（见 ADR-0003 待立），**由以岭正式交付拉动**。在那之前路 B 撑 Demo/POC 档；**不要在路 B 上补 per-app 鉴权**（避免与若依重复造）。

### 柱二｜行业规则包（声明式约束）
- **未来态**：可插拔的声明式 YAML（entities / rules[hard|soft] / workflows / compliance_mapping），作为**约束参数**注入生成；版本化、客户可覆写、覆写留痕。
- **现状 🔴**：未建。仅有 compliance 模块的**数据流向审计**（IT 合规，非行业业务规则）；Specification 有 acceptanceScenarios + 资料 provenance 可作挂点。
- **什么时候建**：Phase 3 行业根据地，选定首个行业（以岭门店巡检/GSP）后启动。**注意原则②**：hard 规则要落成**校验器**，不是塞进生成 prompt。

### 柱三｜三层代码生成引擎（模版 + 约束 + AI）
- **未来态**：① 模版匹配（若依 codegen / 预置模版出大头，确定性）② 约束注入（底座 + 行业规则，硬约束）③ AI 补全（**只补业务增量**）。产出标准模块包。
- **现状 🟡**：当前生成 = **单文件 HTML demo**（daisyUI SPA）+ 路 B Prisma 模型，纯 AI 单次生成。**已撞墙**：deepseek-chat 单次输出 ~8K token（~30KB），丰富 demo 装不下（实证：30KB demo 顶死上限，stats/密集批注被挤掉）。
- **什么时候建**：**这是最近期、性价比最高的一柱**。先做**分段/三层生成**（外壳+首屏一次、每页一次、再拼装）绕开 8K 上限——它既解 demo 丰富度，也是未来若依 codegen 的接法。**可在现栈先做（不依赖若依决策）。**

### 柱四｜外部适配器层（万能插排）
- **未来态**：统一 `adapter_spec`（必选 authenticate/health_check/metadata；可选 query/mutate/subscribe）；CRM/OA/ERP/AI/IoT/SaaS/数据源预置适配器；注册→发现（能力目录）→调用→**隔离**（禁止生成代码直连外部，必经网关/适配器）；AI 模型统一调用（多厂商自动切换/计费/安全审核）。
- **现状 🟡**：仅 **LLM Gateway**（AI 模型抽象、AI_MODE 域内/云、多 profile）= ai_model_spec 的一格；delivery-control `SecurityGate`（隔离/越界检测，**观察+留痕**态，本会话接通）。**缺**：通用适配器框架、能力目录、CRM/OA/IoT 适配器。
- **什么时候建**：Phase 4 + 真实集成需求拉动。先以**连接器契约（接口 + mock + 真实适配器）**形态做最高频的 1–2 个（语音转写 / 一个 CRM-OA），mock 让 Demo 先跑通；正式对接走"集成清单 + 人工介入"。

### 柱五｜智能编排 + 多模型对抗 + 过程资产
- **未来态**：Hermes 编排（意图解析/任务规划/工具调度/多 Agent 协作）；生成与审查用**不同模型对抗**；一切决策操作**可回放**。
- **现状 🟢（最吻合、已大体存在）**：Hermes（建任务/派发）+ 传感器 L1/L2/L3 + 交叉验证（可信引擎）+ **deepseek 生成 / 千问审查**（多模型对抗）+ DecisionLog（过程资产）+ delivery-control 三件套（已接主流程，观察态）。
- **什么时候建**：已具备，**增量增强**即可（编排的意图/规划/工具调度做厚、过程资产做全），不重起。

### 四层归位（对照交底）
- 用户交互层（创造/监护双模式）：Create 🟢、Studio 🟡（薄，缺技术监护视图 + 透明度 L1/L2/L3 界面分级 + businessName/technicalPlan 映射）。
- 智能编排层（Hermes）：🟢。
- 核心能力层（底座/规则包/生成/适配器）：见柱一–四。
- 基础设施层：🟢 Docker/PostgreSQL/**BullMQ-Redis**（交底写 RabbitMQ，无关紧要）/MinIO。

## 矛盾与取舍（对原交底的修正记录）

1. **"hard enforcement 靠 AI" 自相矛盾** → 改为**靠确定性 codegen/校验器**（原则②）。这是整份交底最大的张力，其"三层生成"自身已给出解（模版/约束做硬部分，AI 只补软部分）——前提是别让 AI 生成整块。
2. **若依/Java 全栈 × 现有 NestJS 投资** → 若依是**未来态底座**，**控制面（NestJS：需求/规格/编排/传感器/守护）保留不动**；整份采纳 = 立刻启路 C/Java 大迁移，与"现在先闭环、若依等试点"冲突，故**显式推迟**。
3. **产生产级 Java 模块 × 8K 输出上限** → 单次 LLM 绝无可能；**必须若依 codegen 出 90% + AI 补 10%**（即矛盾 1 的解 / 柱三）。
4. **过度设计风险** → 本 ADR 即对策：当北极星收着，**增量建、需求拉动**（原则④）。
5. **商业模式约束**：私有化 + 守护订阅 = on-prem 订阅许可；**经常性收入锁"再生能力"（生成/守护/模型/更新），不锁运行**；气隙环境用**签名时限许可 + 离线计量对账 + 离线更新包**（详见后续 ADR-0004 私有化授权）。

## Consequences

**变容易：** 每一步有锚点对照（朝北极星走还是偏了/重了）；去重明确（已建的不重做）；对外（白皮书/政企汇报/投资）有一份自洽的目标架构。
**需关注：** 锚点须随真实进展更新（否则又变过时自评）；五柱不可一次性铺（违原则④）。
**将来重访：** 若依底座（ADR-0003）、私有化离线授权与计量（ADR-0004）待立；行业规则包校验器形态、适配器框架细化待真实需求拉动。

## 路标（去重后的建设状态，作为锚点附录）

**A. 已建成 / 已接通（别重做）**
- 工程地基（测试全绿 616+、安全收口、LLM Gateway、BullMQ 部分）。
- 路 B（ADR-0001：数据模型→建表→通用 CRUD→真前端；本会话修资源名大小写）。
- 可信引擎（传感器 L1/L2/L3 + 交叉验证；delivery-control 三件套已接主流程，观察态）。
- 守护中心（巡检 + 健康分 + GuardianCheck + **分级修复** + **月度报告**）。
- 自迭代（持久状态 + 对账 + **防退化护栏**）。
- demo（截图复刻 / 换肤 / 批注 / **暂停显示修复 / 元素级取色 / 编辑解耦 / 售前提示词升级**）。
- 多租户地基（org/membership/tenant-scope/回填）。

**B. 在手 / 部分**
- 双模式：Create 有、Studio 薄；透明度界面分级 + businessName/technicalPlan 映射未建。
- 自迭代/生成丰富度：卡 8K 输出上限。

**C. 待建（下一步，不重复 A）**
1. 守护套餐计费（月报已有、计费没）——守护订阅变现。
2. 验收报告导出（可信交付证据，数据有、成品缺）。
3. **demo 三层/分段生成**（解 8K 上限，柱三的近期落点，现栈可做）。

**D. 未来态 / 暂不建（北极星，等以岭真单子拉动）**
- 若依底座 + 前端/业务-only Java codegen（路 C，ADR-0003）。
- 行业规则包引擎（Phase 3）。
- 通用外部适配器框架（Phase 4）。
- 私有化部署包 + 气隙离线授权/计量（Phase 4，ADR-0004）。
