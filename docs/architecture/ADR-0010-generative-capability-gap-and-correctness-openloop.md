# ADR-0010: AI 生成型需求的能力缺口 + 功能正确性开环 —— 平台能把"名词"摊成 CRUD，产不出"动词"；上线门只验能跑、不验功能对

**Status:** Proposed（2026-06-25；由对真实路B项目"短剧剧本生成平台"跑一次**真实终稿交付**、逐层验交付物反向暴露并形式化）
**Date:** 2026-06-25
**Deciders:** 平台负责人

**关联:** 接 [ADR-0008] 能力来源分流（catalog + maturity 🟢🟡🔴 + gap_workflow + 生成器词汇生长）——本 ADR 指出 ADR-0008 的能力中心**缺"运行时 AI / 生成型后端端点"这一能力类目**，其词汇生长此前只覆盖前端 block（qa/wizard/chart…）。接 [ADR-0009] 上线门——上线门只验"能跑"（编译/部署/契约/冒烟），**不验"功能对"**，本 ADR 指出"功能正确性"对生成型需求是**开环**的，并承接 ADR-0009 未落地的 D1（must 场景运行时证据硬化）。接 [ADR-0005] 自迭代（只改设计态 demoHtml，不碰运行态代码）、[ADR-0007] 契约门。能力设计另见 [ADR-0011] 领域 AI 端点生成。

---

## Context（实证：跑了一次真实交付才看清）

2026-06-25 对真实项目「短剧剧本生成平台」（路B，有 dataModel，backend_kind=crud）发起一次**真实终稿交付**（`deliveryId=00495052-1782363037857`），DeepSeek key 已配（真生成、非模板兜底），逐层翻交付物，实测：

- **DeepSeek 真生成 42 个文件**（后端 23 + 前端 12 + 集成 6），后端模块 `user / script / scene / role / dialogue / favorite`，**全是 Prisma CRUD**。核心证据 `script.service.ts`：
  ```ts
  create(data) { return this.prisma.script.create({ data }); }   // 你 POST 什么就存什么
  ```
  **全后端搜 LLM/AI 调用 = 0 处。** 那条 must 需求「输入大纲 → AI 生成连贯分镜剧本」**无任何运行时实现**——交付物是实体 CRUD 后台，不是生成型产品。
- **上线门据实拦截（诚实在起作用）**：生成代码连编译都没过（Qwen 68 分预警 Prisma schema 截断 / `ScriptRole` model 缺失 / **密码明文存储**；编译闭环 3 轮没修好）→ 上线门判 `build_failed`、不给 URL、不假"已上线"。`goLiveStatus=build_failed`、生命周期 `status` 未被污染（ADR-0009 Slice A 生效）。
- 但这次实测暴露了**两个比"能不能跑"更深的缺口**。

**病灶一句话**：平台能把需求里的"**名词**"摊成 CRUD（对实体增删改查），但**产不出"动词"**——尤其"**运行时调 LLM 的生成型领域端点**"。而上线门只验"能跑"、不验"功能对"；"功能对不对"唯一把关在 D1 验收，D1 对运行时 AI 行为又判不了（demo 阶段只能看静态 HTML → 待人工；交付后也没有"对运行的真端点重验"的机制）。结果：**生成型需求在"功能正确性"这条线上是开环的**——人工在验收页"标记通过"可以放行交付，但交付物证明功能根本没实现（本次实测正是如此：人工放行后，交付后端仍是 CRUD 空壳）。

---

## Decision

### G1｜把"领域 AI 端点生成"作为一等能力缺口，纳入 ADR-0008 能力中心治理
- **能力注册表加类目**：`capability-registry` 增"生成型 / AI 型后端端点（`domain-ai-endpoint`）"能力类目，当前 `maturity=red`。需求命中此类 → `disposeGap` 路由到 **extend-generator**（生成器扩词），**不是 auto-iterate、不是 CRUD 摊平**。
- **生成器词汇向后端生长**：沿用 ADR-0008 给前端加 `qa` block 的复利路径，但落在**后端**——新增"AI 端点模板"：识别生成型功能 → 产出**真调 LLM 的端点**（prompt 模板 + 输入→结构化输出 + 连贯性/约束）。具体设计见 [ADR-0011]。
- **翻绿前诚实标缺口**：在该能力 `maturity` 由 red→green 之前，对这类需求**诚实置 red/deferred + gap_workflow 登记**，绝不悄悄交付成 CRUD 空壳冒充满足。

### G2｜承认"功能正确性"开环，定闭合方向（暂不强行实现，对齐 ADR-0009 D1）
- **重申上线门边界**：上线门验"能跑"不验"功能对"（ADR-0009 已定，不变）。功能对靠 D1 验收。
- **D1 硬化方向**：验收要能对**运行的交付后端**打真实端点——对生成型功能，给**样例大纲**断言"真产出非空 script + 分镜、且与大纲相关"，把"功能对"从"看 demo / 人工拍板"升级为"**对运行态真验**"。这是闭合开环的关键，承接 ADR-0009 Action Item #8（must 场景运行时证据硬化），列为本 ADR action item。

---

## 边界（这条纪律管什么、不管什么）

- **管**：把"生成型 / AI 型需求"作为**显式能力类目**治理（诚实标缺口 + 生成器后端词汇生长路径）；指出并定下"功能正确性开环"的闭合方向。
- **不管**：不在本 ADR 落地具体生成器实现（见 [ADR-0011] 设计）；不立刻硬化 D1（工程量大、分期）；不改上线门"只验能跑"的定位。

---

## Consequences

**变诚实**：生成型需求不再被悄悄交付成 CRUD 空壳冒充——命中即标 red、登记 gap，客户侧如实告知"此能力建设中"。**通过率短期下降**：更多生成型项目会卡在"能力未就绪 / build_failed / D1 未达"——这是对的，它们本就交付不了。**复利**：每补一类 AI 端点模板，所有同型项目永久受益（同 ADR-0008 的 qa block）。**将来重访**：D1 对运行态功能验收硬化后，上线门（能跑）+ 功能验收（功能对）双门齐全，"AI 生成类"需求才算可信交付。

---

## Action Items

1. [ ] **（能力中心）** `capability-registry` 增 `domain-ai-endpoint` 类目，`maturity=red`；`disposeGap` 命中 → extend-generator（不进 auto-iterate）。
2. [ ] **（设计）** [ADR-0011] 领域 AI 端点生成能力：识别 → AI 端点模板 → 真调 LLM → 结构化输出 + 约束 → 契约登记。
3. [ ] **（首例）** 把短剧"剧本生成"端点作为第一个 `domain-ai-endpoint` 模板落地，验 maturity red→green 的复利路径。
4. [ ] **（承接 ADR-0009 #8 · 闭合功能正确性开环）** D1 验收对**运行的交付后端**打真实端点；生成型功能给样例输入断言真产出（非空 + 相关）。
5. [ ] **（生成质量）** 把 Qwen 的结构性发现（schema 截断 / 缺 model / 明文密码等）接进编译/重生成回路；提升生成完整性，降 build_failed 率。
6. [ ] **（客户侧 · 承接 ADR-0008 #6）** 命中 red 能力的需求，前端按 gap 渲成"该能力建设中 / 已登记"动作卡，不暴露内部术语。
