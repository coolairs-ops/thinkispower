# ADR-0015: 终态验收 —— 把门推到"用户能不能用"那一层（功能正确性 + 前端落地两个开环的统一解）

**Status:** Proposed（2026-06-25；把"功能正确性开环"[ADR-0010] 与"前端落地最后一公里"统一为同一根因——验收测量点不够靠后——并给两个终态传感器的打法）
**Date:** 2026-06-25
**Deciders:** 平台负责人

**关联:** 承 [ADR-0009] 上线门（只验"能跑"，本 ADR 补"功能对/界面在"）、[ADR-0010] D1 运行时验收硬化（功能正确性开环）、[ADR-0012] 若依控制台前端、spine-slice-01 的"前端页真落地可见"判据。本 ADR 把这几处零散补丁收成一条原则。

---

## Context（两个开环，同一个根因）

本会话逐层核查暴露两个"门是绿的、用户却用不了"的开环：

- **功能正确性开环**（ADR-0010 实证）：短剧平台真交付，后端 0 处 LLM 调用、是 CRUD 空壳，"一键生成剧本" must 需求无运行时实现；但上线门绿、人工标通过即放行。门验了"能跑"，**没验"功能对"**。
- **前端落地最后一公里**：若依路后端 + 菜单 + 权限都 seed 了，但 codegen 的 vue 未落 plus-ui（`uiRoot` 未接 config）+ 无前端 rebuild 步 → 控制台菜单点进去空页/404。门验了"后端 API 通"（backend-smoke），**没验"前端页面在"**。

**病灶一句话**：两者不是两个 bug，是**同一个病**——**验收/上线门的测量点停在"中间产物"（编译通过、API 探活），没顶到"用户实际能不能用"的终态**。门测的东西确实绿了，但门测的不是用户体验那一层。控制论上：**那一层没有传感器 = 没有反馈 = 开环**。

---

## Decision

### G1｜原则：门的测量点推到"终端体验"，不停在中间产物
凡是"上线门/验收"声称"完成"的维度，**断言点必须落在用户实际接触的终态**，而非链条中段。中间产物（编译过、部署起、API 通）是**前置条件**，不是"可交付"的证据。**可交付 = 用户打开能用、且产出对。**

### G2｜终态传感器一：功能正确性（断言"功能对"）
对**运行的交付后端**打真实端点，喂样例输入、断言真业务产出：
- 生成型/动词需求（如"剧本生成"）：给样例大纲 → 断言**返回非空 + 结构符合（含分镜字段）+ 与输入相关**（非空壳、非 echo）。
- CRUD/名词需求：建一条 → 读回 → 断言**字段一致 + 数据权限隔离真生效**（普通用户只见自己的）。
- 落点：承接 ADR-0010 D1 硬化 + ADR-0009 Action #8（must 场景运行时证据）。**判据进上线门：功能断言不过 → 不给"completed"。**

### G3｜终态传感器二：前端落地（断言"界面在且能用"）
对 `productionUrl` 做**真浏览器级**断言（headless）：
- 页面**真渲染**（非空页/404/白屏）、目标菜单**点得开**、关键页 DOM 含真实数据表格/表单。
- 一条**端到端交互**：登录 → 进目标菜单 → 提交一次表单 → 列表出现该记录。
- 落点：升级 spine-slice"前端页真落地可见"判据为可执行断言。**判据进上线门：前端断言不过 → 不给"completed"。**

### G4｜诚实声明：门是"必要非充分"——倒逼能力，不替代能力（防被当银弹）
终态传感器**测量，不制造**：它让空壳/空页当场现形、停止假成功，并**倒逼**去补能力；但它**不会自动生成动词端点、也不会自动建前端部署链**。
- 短期效果 = **通过率下降**（本就交付不了的项目据实卡住）——这是对的、是诚实在起作用。
- 真正的"修复"仍是分别的工程活：动词端点（ADR-0011）、前端落地链（uiRoot 接 config + rebuild 步）。**本 ADR 只保证"坏的发不出门 + 倒逼去修"，不保证"门一装就能交付"。**

### G5｜怎么判：两角色 + 三锚 + 边界才升级（模型数 ≠ 可信度）

G2/G3 的"断言相关/断言观感"用 LLM 当裁判落地，但**判得可信靠的不是模型数量，是三根锚**——堆模型只增成本，还可能"几个模型一起看走眼 = 三倍自信的错判"。

**只有两个 LLM 角色（不需要第三个模型）**：
- **被测系统内部的运行时 LLM**＝交付系统自己端点调的模型（如生成端点的 DeepSeek）——它是**被测对象**，不是验证器。
- **裁判**＝Qwen（复用现有交叉验证器）——判被测系统产出对不对。
- **"再加一个模型生成参考答案来比对"是多余且更噪**：别拿一个机器答案当另一个机器答案的标尺。

**三根锚（只有一根是模型）**：
1. **锚一·确定性探针（最硬，无模型）**：打真端点 / 真浏览器 + 确定性断言（非空、结构、非 echo / 渲染、可点、端到端往返）。**空壳、空页 100% 死在这根锚**——根本不经过模型。这才是"坐实"的部分。
2. **锚二·裁判独立性（一个模型足矣）**：裁判模型 ≠ 被测系统内部用的模型（DeepSeek 生成 → Qwen 判）。独立性靠"不同源"，**不靠"多请几个"**；第三个不增独立、只增冗余。
3. **锚三·人工黄金校准（地面真值是人，不是模型）**：每类需求备人标黄金样例 + 阈值（ADR-0014 R3/R4），裁判拿黄金校准，非凭感觉。

**边界才升级（省成本）**：单裁判打分**贴近阈值**（疑案）才升级到第二个不同厂商裁判复核；两者仍不一致 → **交人**。平时只跑一个裁判，第二个模型只花在边界，不花在每一次。

**诚实底线**：灾难层（空壳/空页）由确定性锚 100% 坐实；语义/观感层是**高置信、非确定**，由"独立裁判 + 人工黄金 + 疑案升级"逼近，不号称 100%。**能用确定性判的别用裁判；裁判一个独立的够；疑案才请第二个、最后归人。**（前端 G3 同构：确定性浏览器断言是锚，VL 视觉模型是观感裁判，疑图才升级/交人。）

---

## 边界（管什么、不管什么）

- **管**：定"测量点推到终态"原则；定两个终态传感器的断言打法；定判法（两角色 + 三锚 + 疑案升级，G5）；把判据接进上线门；声明门必要非充分、模型数≠可信度。
- **不管**：不实现动词端点生成能力（ADR-0011）；不实现前端落地链本身（spine-slice/部署工程）；不改上线门"确定性二值"的机制（只是新增两个断言项）；不引第三个模型做常态验证（只在疑案升级时用第二裁判）。

---

## Consequences

**假成功被堵死**：上线门绿 = 用户真能用 + 功能真对，"出空货还报喜"这条路关闭——直接加固核心卖点"可信交付"。**完成度诚实化**：当前 40–55% 的真实交付完成度会被如实暴露（不再被"后端绿"掩盖），短期通过率下降。**倒逼闭环**：两个开环从"看不见"变"看得见的红灯"，逼能力缺口被真补，而非蒙混。**与纪律一致**：延续"靠门不靠自评"，把门从中间产物推到终端。

**代价**：终态断言比中间产物检查**更重**（要起真后端打端点、起 headless 浏览器点页面），CI/交付时延上升；样例输入/期望产出需人工标注（同 ADR-0014 R3 黄金用例）。

## Action Items

1. [ ] **（G2·功能）** 上线门加"功能断言"项：对运行后端打样例端点，生成型断言非空+相关、CRUD 断言增删改查+数据权限隔离。承接 ADR-0010 D1 / ADR-0009 #8。
2. [ ] **（G3·前端）** 上线门加"前端断言"项：headless 打开 productionUrl，断言渲染+菜单可开+一条端到端交互。升级 spine-slice 判据为可执行。
3. [ ] **（接门）** 两项断言不过 → `goLiveStatus != completed`（接 ADR-0009 二值闸，新增断言项，不改机制）。
4. [ ] **（人工·黄金用例）** 每类需求备样例输入 + 期望产出（人工标注，复用 ADR-0014 R3/R4 eval 体系）。
5. [ ] **（判法·G5）** 裁判用 Qwen（≠ 被测系统内部模型）复用现有交叉验证器；确定性断言为主、裁判为辅；疑案（贴阈值）才升级第二厂商裁判、再不一致交人。**不引第三个模型做常态验证。**
6. [ ] **（诚实声明·防银弹/防堆模型）** 文档/对外口径写明：终态门倒逼能力、不替代能力；通过率短期下降是预期；**模型数量 ≠ 可信度，坐实靠确定性锚 + 人工黄金**。
7. [ ] **（时机）** 随客户系统脊柱推进落地——G3 前端断言并进 spine-slice Phase 3/4；G2 功能断言对客户系统先做 CRUD+数据权限断言，动词端点断言待 ADR-0011。

---

## 时机

不单独立项抢跑——**随脊柱走**：G3（前端断言）正是 spine-slice Phase 3/4"真人走一遍"的可执行化，现在就该并进判据；G2（功能断言）对客户系统先落"CRUD + 数据权限隔离"断言（脊柱范围内），"动词端点真产出"断言等 ADR-0011 能力就位再加。**先把门装在客户系统这一个真样板上证通，再推广。**
