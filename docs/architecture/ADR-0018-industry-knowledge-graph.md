# ADR-0018: 行业知识图谱建设 —— Hyper-Extract 作录入器，喂给"规则引擎(执行)"与"覆盖式澄清(参照)"两个消费者

**Status:** Proposed（2026-06-29；行业根据地 Phase 3 的内容地基。引入 Hyper-Extract 作行业图谱**离线录入器**，关在适配器后、过人审门、排脊柱+serve 之后。明确"图谱=静态内容"与"运行时规则引擎=执行器"的分工。）
**Date:** 2026-06-29
**Deciders:** 平台负责人

**关联:** 喂 [ADR-0016] 覆盖式澄清（行业基准参照系）；喂现有 `rule-engine`（运行时规则引擎的内容来源）；遵 [ADR-0014] R3 人审门、[ADR-0015] 裁判需人工黄金；接 Phase 3 行业根据地路线（`WHITEPAPER-v2` 价值四/二）。

---

## Context

行业根据地（Phase 3）需要每个行业的**知识图谱/规则库**：领域实体模型、关系、术语表、合规规则、验收场景、常见失败模式。这块在能力清单是 🔴 空白，冷启动最苦——面对一堆行业规范/手册/制度，要人工读出实体/关系/规则。

**关键分工（先讲清，免得混）**：
- **行业知识图谱 = 静态内容**：实体/关系/规则的**数据**。本 ADR 用 Hyper-Extract 把它从行业文档抽出来。
- **运行时规则引擎 = 执行器**（已存在 `app-runtime/rule-engine/`，确定性零 LLM + 证据链 + 业务可改阈值）：运行期对真实数据跑"条件→结论/动作"。
- **两者关系**：图谱**喂**引擎（规则内容）与澄清（行业基准）。图谱是"知道有这条规则"，引擎是"规则在跑着的系统里真命中"。**别把建图谱当成建引擎。**

---

## Decision

### D1｜Hyper-Extract 作行业图谱"离线录入器"（不是运行时依赖）
- 用 `he parse 行业文档 -t industry/xxx` 抽 `entities{name,type,description}` + `relations{source,target,type}` → 喂行业实体模型 + 术语表雏形。
- 借它的 **YAML graph 模板范式**（entities/relations/identifiers/guideline）定义平台自己的行业抽取 schema；其工业/法律/医疗预设作对应行业起点参照（**借设计，不直接用**）。
- **`feed` 增量演进** = 图谱"持续生长"：每交付一个该行业项目，喂新文档/新规则进去，对应平台"经验沉淀"理念的图谱级复利。
- 本地 vLLM 部署、数据不出域（对齐 `AI_MODE=local`）。

### D2｜两个消费者（图谱建好后喂给谁）
1. **喂运行时规则引擎**：抽出的业务规则 → 经人审 → 转成 `rule-pack`（结构化 when/then）→ 现有 `RuleEngineService` 运行期执行。
2. **喂 ADR-0016 覆盖式澄清当"行业基准参照系"**（**最大增强点**）：通过 `he-mcp` 的 `ask/search` 暴露图谱；ADR-0016 第③步缺口扫描查图谱，把完备度从"通用槽位"升级到"**行业基准槽位**"——系统能判"这个行业该有的实体/关系/规则你提到了没"，自动补成选择题。
   - 例：业务人说"药店巡检"，查图谱发现 GSP 规范通常含温湿度/效期预警/近效期处置 → 自动问出来。**这是行业图谱 × 覆盖式澄清的乘法效应，是平台独有的。**

### D3｜诚实边界（沿用既定纪律）
- **建图谱 ≠ 建运行时引擎**：Hyper-Extract 只产静态内容；执行交现有 `rule-engine`。
- **概率输出必过人审门**：抽取产物入行业包前由领域专家审核（ADR-0014 R3）；它是录入助手，不是真值源。
- **早期 Python 工具 → 进程外、关适配器后**：CLI 建库 / MCP 查询，不焊生产关键路径；定义 `KnowledgeIngestor` 接口，Hyper-Extract 是其一实现，可换。
- **丢掉**：超图/时空图/引擎动物园/Obsidian——只取朴素 entities+relations 图谱 + feed + MCP。

---

## 边界（管什么、不管什么）

- **管**：行业图谱的**录入**（Hyper-Extract CLI）、**查询出口**（MCP 接 ADR-0016）、与规则引擎/澄清的喂养关系、人审门、适配器隔离。
- **不管**：不建运行时规则引擎（已存在，另见其 action/trigger 增量）；不实现 Hyper-Extract 深层抽取算法（commodity，借用不重写）；不碰全域数据中台（已封存 `VISION-PARKING`）。

---

## Consequences

**冷启动有了录入器**：行业图谱从"人工读资料"变"抽取→人审"，且 `feed` 让它随交付复利生长。**乘法效应**：图谱接进覆盖式澄清 = 行业基准完备度，别人抄不走。**保持纪律**：建的是内容、执行交确定性引擎；输出过人审；工具关适配器后、可换。

**代价/风险**：抽取产物需领域专家审核（人力）；MCP 接 ADR-0016 增运行期依赖（Hyper-Extract 服务要在）——可降级为"无行业基准时退回通用槽位覆盖度"，不硬依赖；行业 schema 与 `app-spec.types.ts` 槽位要对齐映射。

---

## Action Items（Phase 3，排脊柱+serve 之后）

1. [ ] **（接口先行）** 定义 `KnowledgeIngestor` 接口（输入文档→输出 entities/relations，中立 schema）；Hyper-Extract 作首个实现，关适配器后。
2. [ ] **（建库 CLI 路）** `he parse 行业资料 → 适配器映射成行业包 schema → 人审 → 入库`；借 YAML 模板范式定义行业抽取 schema。
3. [ ] **（查询 MCP 路 · 最大增强）** `he-mcp` 暴露 ask/search；ADR-0016 覆盖式澄清第③步接行业图谱当基准参照系（无图谱时降级通用槽位）。
4. [ ] **（喂规则引擎）** 抽出的规则经人审转 `rule-pack`（when/then）→ 现有 `RuleEngineService` 执行。
5. [ ] **（人审门）** 抽取产物入行业包前领域专家审核（ADR-0014 R3）。
6. [ ] **（不做）** 不建运行时引擎、不重写抽取算法、不碰全域数据中台。
