# ADR-0016: 一项目一若依租户硬隔离 —— 激活若依-Plus SaaS 多租户，根治"共享控制台张冠李戴"

**Status:** Proposed（草案，2026-06-29；待平台负责人拍板。本会话密集踩坑后逼出——"demo↔交付不是一个产品"的一整类 bug 的总根源）
**Date:** 2026-06-29
**Deciders:** 平台负责人（待）

**关联:** 承接 [ADR-0012]（若依统一控制台 + 权限分身）——0012 解决"一套界面按角色分身"，本 ADR 解决"**不同项目之间**"的硬隔离；与"控制台 serve 基建自动化"（候选②）相邻（per-tenant URL 是其后续）；取代/收编本会话的软隔离补丁（`a082264` 账号引导 / `b69e94a` 重交自愈种账号 / `4bac241` 菜单改挂）。

---

## Context（为什么现在定）

ADR-0012 落地后，所有 ruoyi 项目交付到**同一个若依实例 + 同一个租户 `000000` + 同一个 plus-ui 控制台（8089）**，`productionUrl` 对所有项目是同一地址。靠"项目专属角色 + data_scope + 项目域菜单"做**软隔离**。

本会话密集踩坑证明软隔离扛不住——"客户系统的上线产品里混进设备维修菜单"这一整类"张冠李戴"问题，根因全在共享：
- **超管穿透**：若依超管（admin，租户 000000）无视角色看到**所有项目**的菜单/数据。我们只能加"勿用 admin"引导（`a082264`），堵不住。
- **数据混库**：所有项目业务数据 `tenant_id` 都是 `000000`，挤在同一批物理表里，没有库/租户级硬隔离，只靠 data_scope 行级软挡（一旦角色配错或是 admin 就裸奔）。
- **菜单/账号脆**：旧项目菜单挂错父目录、缺专属账号——本会话连修三个补丁（自愈种账号、菜单改挂）才勉强让"用专属账号登只见本项目"成立。
- **无运营边界**：没法按项目/客户独立计费、退租、导出、停服；一个项目置备炸了拖垮整个共享控制台。

**关键勘探发现（决定了本 ADR 的可行性与成本）**：若依-Plus 的 **SaaS 多租户机器其实已经装好且开着**——
- `sys_tenant` 表存在（当前只有 `000000`）；`tenant.enable: true` 已开启；
- **所有业务表已带 `tenant_id` 列**（ruoyi-ddl 基础列），拦截器会自动 `WHERE tenant_id=<当前租户>`；
- 接缝只有一处：平台 `RuoyiClient.login` 发的 `tenantId: cfg.tenantId` 是**单值 000000**，置备也全落 000000。

**一句话病灶**：不是缺多租户能力，而是**所有项目都被塞进同一个租户**，把已装好的隔离能力空跑成单租户。

---

## Decision（草案·五条）

### D1｜一项目（或一客户）一若依租户
置备时为项目分配**独立 `tenant_id`**（新建 `sys_tenant` + 租户套餐 `sys_tenant_package` + 该租户的管理员账号）。`tenantId` 从平台单值 `000000` 改为**按项目派生/存储**（写进 `backendRuntime.tenantId`）。

### D2｜数据隔离：靠租户拦截器（地基已就位，近零 DDL 改动）
业务表已带 `tenant_id`、`tenant.enable=true`。每项目数据自动按其 `tenant_id` 隔离（`WHERE tenant_id`）。**这是本方案最大红利：数据硬隔离几乎免费。**

### D3｜菜单/功能隔离：靠租户套餐（**关键，非 tenant_id**）
`sys_menu` 在租户排除表里（菜单全局共享、不按 tenant_id 过滤）。隔离靠 **`sys_tenant_package`**：每项目租户绑一个套餐 = **只含本项目的业务菜单**（+ 必要系统菜单）。租户内的管理员**也只看到套餐内菜单**——这才真正堵住"超管/管理员看到所有项目菜单"，无需再靠"勿用 admin"引导。

### D4｜RBAC/账号落到租户内
`seedRoles/seedMenusAndGrant/seedUsers` 在**目标租户上下文**内执行；登录用**本项目租户 ID**（接缝：`RuoyiClient.login` 的 tenantId 改为按项目）。`ruoyi-data-proxy`（终端用户代持 token）也按项目租户登录。

### D5｜控制台仍共享一套 UI，租户在登录态区分
plus-ui SPA 是通用的，**租户由登录决定**；不要求一项目一前端实例。`productionUrl` 可短期仍为共享 8089 + 租户登录，长期 per-tenant 路由/子域（归入候选②"serve 基建"，本 ADR 不强求）。

## Consequences

**正向：**
- 数据**行级硬隔离**自动生效（拦截器 + 已有 tenant_id 列），成本极低。
- 用户/管理员**按租户套餐**真隔离，超管穿透问题根治；本会话三个软隔离补丁（账号引导/自愈种账号/菜单改挂）可逐步被租户套餐收编。
- 解锁运营边界：按租户**计费 / 退租 / 独立导出 / 独立停服**；单项目置备故障不再天然拖垮他人（仍同实例，但数据/账号隔离）。

**代价 / 限制（必须正视）：**
- **⚠️ 共享 schema 是命门**：若依-Plus SaaS = **共享库共享表**多租户，租户隔离的是**行**、不是 **schema**。两个项目若都有同名实体（如都叫 `contract`），**共用同一张物理表 + 同一套列**——租户只分行，**列结构冲突仍在**（A 项目的 contract 字段 ≠ B 项目的，后置备会改/撞表）。**这是本方案不解决的部分**，必须配套（见开放问题 Q1：按项目给表名加域前缀）。
- **租户生命周期**要管：建/停/删租户、租户套餐随项目菜单生成。
- **存量迁移**：现有项目数据全是 `tenant_id=000000`，要么重置备进新租户、要么数据级 re-tag（涉及历史行 + 账号 + 菜单授权）。
- 控制台仍是**一套共享 UI**（非一项目一部署）——可接受，但 per-tenant URL/子域是后续。

---

## Alternatives considered

- **A. 一项目一若依实例（独立容器/库）**：隔离最强（连 schema 都隔），但 N 容器 / N 编译 / 重运维，成本高。→ 留给高价值私有化客户，SaaS 过重。
- **B. 一项目一 schema/库（同实例多数据源）**：若依-Plus 有动态数据源，但业务 codegen 面向单 schema，改造复杂。
- **C. 维持现状软隔离（本会话补丁）**：超管裸看、数据混库、无退租边界。→ 作为**过渡可以**，作为终态**拒绝**。
- **D（选中）一项目一租户（激活已装的 SaaS 多租户）**：复用现成机器，行+用户+菜单真隔离，增量成本最低。**代价=共享 schema、同名实体冲突**，靠 Q1 配套。

## 开放问题（拍板/设计前要定）

- **Q1（最重要）同名实体跨项目冲突**：共享 schema 下两个项目的 `contract` 撞表。是否**按项目给业务表名加域前缀**（如 `p12acf9f0_contract`）？这会连带改 codegen 的 tableName 派生、菜单 perms、vue 组件路径——影响面不小，但不解决就还会撞。
- **Q2 租户粒度**：一项目一租户，还是**一客户一租户**（一个客户名下多个系统共一租户、多套餐/多菜单）？后者更贴"一个客户=一个空间"的运营直觉，也省租户数。
- **Q3 存量迁移**：ed541b1e/合同系统等 000000 项目怎么迁——重置备到新租户（干净但要重编译）还是数据 re-tag（快但要迁历史行/账号/授权）？
- **Q4 软隔离补丁去留**：租户套餐上线后，本会话的"项目专属角色 + 菜单改挂 + 账号引导"哪些被收编、哪些保留兜底？
- **Q5 productionUrl**：短期共享 8089 + 租户登录够不够？per-tenant 子域/路由归并到候选② serve 基建一起做？

---

## 分期落地建议（待批准后才写码）

- **P0 设计收口**：定 Q1（表名是否加域前缀）+ Q2（租户粒度）——这两条决定整个改造形状，必须先定。
- **P1 置备链接租户**：provisionApp 建 `sys_tenant` + 套餐 + 租户管理员；`tenantId` 全链路按项目（config→login→seedRoles/Menus/Users→data-proxy）；descriptor 存 `tenantId`。
- **P2 菜单/功能隔离换租户套餐**：套餐 = 项目菜单白名单；验证"租户内管理员只见本项目"。
- **P3 存量迁移**：按 Q3 决议迁 000000 项目。
- **P4（可选，并入②）**：per-tenant productionUrl / serve 基建。

每期独立可验、可回退；P1 前不动存量运行态。

---

## 验收口径（怎么算"硬隔离成立"）

确定性二值，不靠感觉：
1. 项目 A 的终端用户（含 A 租户管理员）登录控制台 → **只见 A 的菜单、只查到 A 的数据行**；切到 B 看不到 A。
2. 跨租户直接调 B 的资源（带 A 的 token）→ 拿不到 B 的行（拦截器 WHERE tenant_id 生效）。
3. 删/停 A 租户 → A 的数据/账号/菜单整体失效，不影响 B。
4. 全新两个同类项目（都含 `contract`）端到端交付 → 各自独占、互不串（验 Q1 的表名策略是否真解决撞表）。
