# ADR-0012: 政企后台 = 若依统一控制台 + 权限分身 —— "一个界面 vs 两个界面"的路由规则，停止"平台自造前端"摇摆

**Status:** Accepted（2026-06-25；由本轮"工程控制论体检 + 若依脊柱实测"逐步逼出，平台负责人拍板"咬牙走这条、先做一个确定能打通的产品"）
**Date:** 2026-06-25
**Deciders:** 平台负责人

**关联:** 落实 [ADR-0003] 强制底座=若依（把"底座"从"后端"扩到"**整套控制台**"）；终结 [ADR-0010]/[ADR-0011] 暴露的"平台自造前端"不确定性来源；与 [ADR-0009] ③（验收/上线门对若依分流）配套；与 [ADR-0008] 能力路由同构（按形态选路）。视觉沿用「内置 6 套政企主题」[[project_template_library]]，映射为若依主题预设。

---

## Context（为什么现在定）

本会话逐环节实测后收敛出一个硬事实：**"平台自造前端"是交付不确定性的最大单一来源**——空壳 demo、缺表单、缺登录、字段漂移、验证器判静态 HTML 的假阳/假阴，几乎都出自"平台并行造一套前端"。而**若依的代码生成器从同一 `gen_table` 同时产后端 + 前端页**，前后端一致是构造性的；其 RBAC + **数据范围（数据权限：全部/本部门/仅本人）** 原生支持"同一套界面、按角色分身"。

平台负责人借鉴同类产品（政企 AI 应用平台）后提出关键简化：**"前端用户 vs 后端管理员"不是两套界面，而是同一套控制台 + 权限区分身份**——这正是若依原生模型。

**病灶一句话**：之前默认"前台 + 后台两套界面"，对**纯内部业务系统**是徒增一倍复杂度，且把"平台自造前端"的全部难题拉回来。多数政企系统其实只需要**一套登录后的控制台**，角色差别只是"看多少、能改谁的数据"。

---

## Decision（五条）

### D1｜路由规则：一个界面 vs 两个界面（核心，写死，不再逐项目纠结）
判据只有一条——**用户里有没有"不该看到控制台/后台形态"的一类人**：
- **内部业务系统**（全是登录后干活的角色，差别仅权限/数据范围）→ **一个若依统一控制台 + 权限分身**。CRM/进销存/OA/政企内部系统属此。
- **有对外 / 匿名 / 品牌化前台需求**（公众办事、消费者下单、营销页、免登录自助）→ **两个界面**：若依做后台 + 平台自造前台（接受其更高不确定性，仅在确有此需求时付这个代价）。

默认走"一个界面"；只有命中"对外/匿名/品牌化"才升级为两个界面。

### D2｜权限分身：身份靠角色 + 数据范围，不靠"分两套页"
- **角色权限**控制菜单/按钮显隐（普通用户看不到系统管理/用户管理）。
- **数据范围**（若依数据权限：全部/本部门/仅本人）控制能看/改谁的数据。普通用户=仅本人、管理员=全部，**一处配置、全局生效**，靠若依 `create_by`/部门自动过滤。
- 平台职责：从需求**生成 dataModel + 角色/数据范围矩阵** → 驱动若依 codegen 出控制台 + 配 RBAC。**不再生成第二套前端、不再自写数据权限。**

### D3｜视觉：6 套主题保留，映射为若依主题预设（不是像素级复刻）
- 内置 6 套（政务蓝/党政红/深色指挥大屏/商务墨绿/简约企业白/科技紫蓝）**保留为可选**，映射为 **6 个若依主题预设 = 主题色 + 明暗模式 + 侧边栏深浅**，置备时按所选预设配置若依实例。
- **诚实边界**：若依是 Element-Plus，换肤 = **同色系重着色 + 明暗**，得到"若依被染成该色系"，**非 daisyUI 专属视觉的像素复刻**；"**大屏**"是另一种数据可视化布局，**不在若依标准控制台范围**——需要大屏时走"对外/大屏"轨（D1 的两个界面分支）。
- 即：**色系/明暗可换；任意版式/截图复刻/大屏不在一个界面这条轨内。**

### D4｜客户系统 = 首个"一个界面"样板
- 客户系统（纯内部 CRM）按 D1 = **一个若依控制台**。其缺口（数据看板/详情编辑/客户-项目关联/**数据权限**）由若依原生满足：数据权限=数据范围配置；详情/关联=若依生成页；看板=若依首页/计数（聚合看板若需图表，按 ADR-0008 后续）。
- 验收目标：配 `超级管理员(全部数据)` + `普通用户(仅本人)` 两角色，分别登录验证"普通用户只见自己建的客户、管理员见全部"→ 一套界面 + 权限分身**真打通一个能上线的产品**。

### D5｜"平台自造前端"退场（限定保留）
- 对**内部业务系统**，平台自造前端（daisyUI 块系统）**退出主交付**——它造成的不确定性不再为这类项目承担。
- 仅保留给：D1 的"对外/品牌化前台"分支、以及生成型 AI 端点的定制界面（ADR-0011）。本会话已建的"表单/登录门生成"（ADR-0009 ①）作为该保留轨的能力，不浪费。

---

## 边界（管什么、不管什么）

- **管**：把"一套控制台 vs 两套界面"按用户形态写死路由；把身份区分统一到若依 RBAC + 数据范围；视觉用 6 主题预设换肤。
- **不管**：不追求若依控制台做任意视觉定制（接受 Element-Plus 重着色）；不解决"大屏/对外品牌前台"（那是两个界面分支的事）；不改 ADR-0011 生成型 AI 端点方向。

---

## Consequences

**确定性 / 成功率台阶式上升**：内部系统交付 = 用成熟若依生成器，前后端构造性一致、权限/用户管理开箱，去掉平台自造前端的整类不确定性。**维护面收缩**：不再维护两套 UI + 自写数据权限。**叙事更强**：对政企"用成熟底座自动配好一套带权限的控制台"比"AI 现编前端"更可信。

**代价（认清）**：视觉受限于若依 Element-Plus 重着色（非任意定制/复刻）；技术栈锁 Java/Spring + Vue；不适合对外/匿名/大屏场景（那走两个界面分支）。

**平台价值上移**：从"AI 现编前端"变为"从需求自动生成 dataModel + 角色/数据范围矩阵 → 驱动若依产出整套控制台"。

## Action Items

1. [ ] **（样板）** 客户系统按 D1 切若依统一控制台；配 `超级管理员(全部)` + `普通用户(仅本人)` 两角色，登录验证数据范围分身。
2. [ ] **（视觉）** 6 套主题 → 若依主题预设（主题色 + 明暗）映射表；置备时按所选预设配置若依实例默认主题。
3. [ ] **（生成）** 平台从需求产出"角色/数据范围矩阵"，驱动若依 RBAC + 数据权限配置（替代自写过滤）。
4. [ ] **（上线门）** `productionDeliver` 对若依项目识别 → 据 backendReady 判 `goLiveStatus=completed`（承接 ADR-0009 ③ 未完的形式门）。
5. [ ] **（路由落地）** 需求/规格阶段判定"内部系统 vs 对外前台"，决定一个界面 / 两个界面；默认一个界面。
6. [ ] **（保留轨）** 平台自造前端能力（表单/登录门生成）归档到"对外前台 / AI 端点"轨，不用于内部系统主交付。
