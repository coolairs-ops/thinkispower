# ADR-0003: 生成系统强制底座 + 若依-Plus 作首个实现

**Status:** Accepted（产品方向硬决策；由产品自身拉动，不再等单一客户）
**Date:** 2026-06-20
**Deciders:** 平台负责人（产品方向裁定）
**关联:** 落实 [ADR-0002] 柱一（统一底座/强制内核）+ 柱三（三层生成）+ 原则①②；推广 [ADR-0001] 的 `BackendRuntime` 可替换接口为"底座适配器"；是自迭代诊断"demo 修不出权限/级联/审计"的结构性解。**显式覆盖 ADR-0002 原则④对本决策的"等以岭拉动"门槛**——见「对 ADR-0002 的修订」。

---

## Context（为什么现在就定，不等客户）

- **自迭代实证**：放任生成自由 HTML/CRUD → 输出空间无界 → 单次 LLM 想不全每个环节 → 多用户权限/角色数据隔离/级联删除/审计「修不出」→ 回路卡死。**可靠性的命门是收窄输出空间。**
- **定位**：平台面向政企/央国企内部管理系统，命门是「可信交付」。这类系统约 80% 是 RBAC + CRUD + 审批 + 报表 + 审计——**正是若依的甜区，政企本就认、信创本就适配。**
- **裁定**：产品方负责人定性——这是**产品的地基方向，不是投机性功能**；地基该在有客户前就立稳，故**不沿用 ADR-0002 原则④「等真实单子兜底」的门槛**，本决策由产品自身拉动。

---

## Decision（两条写死）

### D1｜生成系统强制走"强制底座"，禁止自由 HTML

- 思想动力生成的交付物，**不再是自由发挥的单文件 HTML / 任意 CRUD**，而是**面向"强制底座"的标准模块**：实体模型 + 底座槽位配置 + 仅业务增量。
- "强制底座"是一份**契约/接口**（沿用 ADR-0001 `BackendRuntime` 的可替换思想，泛化为"底座适配器"）：权限/审计/审批/字典/动态菜单等**强制内核由底座提供，生成的应用禁止自建**（ADR-0002 柱一原文）。
- hard enforcement 靠**确定性 codegen + 校验门驳回不合规产物**，不靠提示词（ADR-0002 原则②）。生成进的是已知框架的槽 → 测试门才校验得真（呼应 ADR-0005「门真才站得住」）。

### D2｜若依-Plus 作"强制底座"的首个、且政企主力实现

- **选型理由（市场契合 > 纯技术）：**
  - **MIT 授权** — 可私有化/闭源/任意修改，契合私有化交付与「断供锁再生不锁运行」（对比 JeecgBoot 等授权更严，闭源 SaaS 要顾忌商用条款）。
  - **信创生态** — 达梦/人大金仓/麒麟/鲲鹏适配，央国企硬要求。
  - **自带 codegen** — 模版确定性出 ~90% + AI 只补 ~10% 业务增量，正是柱三解法，且绕开单次 LLM ~8K 输出上限。
  - 政企采购/安审本就认、社区参考海量。
- **底座做成接口、若依作第一实现**：锁定"必须走底座"这条铁律，但不把全链焊死在若依一家——日后某客户要别的底座（.NET/国产中台等）换适配器即可。

---

## 边界（这条铁律在收窄定位，且这是好事）

| 在范围（若依甜区） | 不在范围（强制若依会别扭） |
|---|---|
| B 端政企/央国企管理系统：台账/审批/CRM/巡检/报表/RBAC/工作流 | C 端消费应用、营销/内容站、实时/IoT、游戏、高度定制 UI、移动优先 |

- 不在范围的形态**不在思想动力强制底座的生成范围**；要做须另立底座实现（接口已留），不在本 ADR。
- **收窄 = 聚焦，不是损失**：把战场收敛到能可靠交付、客户又认的那块。

---

## 对 ADR-0002 的修订

- **柱一"什么时候建"**：原「由以岭正式交付拉动」→ 改为**由产品方向拉动，即刻确立为地基方向**（实施仍渐进、不砸现有闭环，见下）。
- **原则④**：仍适用于行业规则包/适配器框架/私有化授权等其他重投入；**唯本决策（强制底座 + 若依）显式豁免其客户门槛**，因其为地基而非投机功能。

---

## 落地次序（锁方向，但不砸现有闭环；渐进）

1. **立约**（本 ADR）：D1/D2 定为铁律——后续所有"生成真应用"的设计都以"产强制底座模块"为目标，不再投资"自由 HTML 真应用"方向。
2. **接口先行**：把 ADR-0001 `BackendRuntime` 泛化为"底座适配器"契约（数据模型 + 槽位 + 校验门），与现栈解耦。**零 Java。**
3. **若依 adapter 薄实现**：起一个若依-Plus 实例，打通"实体模型 → 若依 codegen → 起得来的管理系统"最小闭环（单实体 CRUD + RBAC 即可）。
4. **生成接管线**：把需求补全（A/D/E + 回写）的产物（实体/页面/角色/数据权限）喂给**若依 codegen**，而非喂给 demo HTML 生成——这是把前面几锤的「诊断/回写」真正接到「产真后端」的桥。
5. **demo 路径降级定位**：分段生成（柱三，现栈）保留为**售前快速预览**；"真交付"走若依底座。两者并存、分工明确。

> 步骤 1-2 零 Java、不动现有闭环；步骤 3 起才引入若依实例。**先锁方向，再按力气推进。**

### 修订（2026-06-20，据实测演进）
- **步骤 2-4 已实测打通**：底座适配器(`BackendRuntime`)、`provisionApp`(实体→真若依后端+RBAC)、需求补全产物→若依 codegen(适配器① IR→AppSpec)。
- **demo 不再只是"售前预览"**：适配器②让 demo/复刻前端 serve 时切若依真数据(同 appData 接口、HTML 不改)，**demo 升级为"经适配器②的真交付前端"**——前端生产力(demo/复刻) + 若依真后端合体。这是对上面第 5 条"demo 降级"的演进。
- **待补的缺口（下一步）**：本 ADR 的"填槽+校验门"只用在**后端 codegen**；前端迭代仍自由生成、对若依契约是盲的 → 迭代撞验收门。需把同一纪律扩到前端迭代(应用契约统一驱动前端生成+后端置备+验收门)，见 ADR-0006 实施进展 + ADR-0007 候选。

---

## Consequences

**变容易：** 生成可靠（填槽而非凭空发明）；政企信任与信创合规直接继承；交付物统一 → 守护可统一巡检/修复；权限/审计/审批「修不出」的结构性难题消失。

**需关注：** 引入 Java/若依栈（与 NestJS 控制面并存，运维多一摊）；codegen 接管线是实打实工程量；输出空间收窄 = 产品定位收窄（须接受「只做政企管理系统」）；这是**先于确认客户的地基投入**（产品方已裁定承担）。

**将来重访：** 若依 codegen 与需求补全产物的映射保真度；双部署（SaaS + 私有化气隙）；非若依底座实现的触发条件；与 ADR-0004（私有化离线授权）合流。
