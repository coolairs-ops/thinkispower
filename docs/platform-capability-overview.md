# Think-is-power 平台能力清单（用于需求适配）

> 用途：拿这份对照外部项目需求，逐条判断"能接 / 半成可补 / 不在能力圈"。
> 成熟度口径：🟢 已端到端通、🟡 有地基/半成、🔴 未起。**按真实代码兑现度标注，非纸面规划。**
> 更新：2026-06-20

---

## 0. 一句话定位

**面向普通人（非技术）的"软件生成 + 交付 + 持续守护"平台**：用户用大白话提需求 → 平台自动产出可运行的业务系统（表单/列表/审批/权限/台账类）→ 上线交付源码 → 上线后定时体检自愈。定位**政企/央国企、数据不出域**（可全程本地大模型 `AI_MODE=local`）。

核心差异点：不是"只生成一个 Demo 给你看"，而是**需求确认 → 真后端 → 交付 → 守护**的完整闭环；且把"可信"做成**校验器/测试门**（硬约束），不是提示词里的口头护栏。

---

## 1. 端到端能力地图

```
① 需求采集与补全 → ② 设计确认(Demo/复刻) → ③ 真后端生成 → ④ 交付上线 → ⑤ 持续守护
   (访谈+补缺口)      (可点的样板/截图复刻)   (数据模型→建表→CRUD)  (源码/部署/链接)  (定时体检+分级自愈)
```

每一段下面拆开讲"解决什么问题"。

---

## 2. 分模块能力（解决什么问题 + 成熟度）

### ① 需求采集与补全 🟢

**解决的问题**：普通用户提的需求天然残缺/口语化，直接生成会"补了一堆没要的、漏了真要的"。

- **结构化需求访谈**：把口语需求整理成结构化 IR（实体/页面/功能/角色/流程）。🟢
- **完备性批判（A）**：LLM 当资深需求分析师，找出"缺的页面/实体/字段/数据权限"等缺口，每条带证据。🟢
- **处置分类（D）**：每个缺口判 `autofill 自动补 / ask 问用户 / info 仅提示`，ask 的出**选择题**（非技术友好）。🟢
- **追加问答窗口**：把所有要问用户的（需求缺口 + 关系 + 业务规则）**合成一屏选择题**，答完自动回写、规格自动重生成。🟢
- **业务规则补全**：从功能描述里抽散落的规则（"调整需上报"=审批），明确的自动抽、模糊的问。🟢
- **实体关系补全**：从已填内容推实体关系并问清——
  - 1—N 主子表（客户 1—N 项目）🟢
  - 自关联/树（部门树、分类树）🟢
  - N—N 多对多（学生选课，合成中间表）🟢
- **回写随动**：补全结果回写进需求 IR + 正式规格（未冻结自动反映、已冻结给"待解冻"信号不偷改）。🟢

> 价值：**把"让客户画 ER 图/写 PRD"降级成"答几道选择题"**，这是面向非技术用户的关键。

### ② 设计确认（生成可点的样板） 🟢

**解决的问题**：需求对不对，光看文字说不清，要给用户一个能点的东西确认。

- **Demo 生成**：需求 → 单文件可交互 HTML（列表/表单/按钮/弹窗 + 真数据自播种），用于确认而非交付。🟢
- **分段生成**：绕开单次 LLM ~8K token 输出上限，确定性外壳 + 每页独立生成 + 并行，支持多页复杂 Demo。🟢
- **截图复刻**：需求文档附蓝湖/原型截图 → 两段式（vision 看图出布局描述 → text 出 daisyUI 页面）复刻，规避长代码退化；人在回路可改 OCR 错。🟢
- **外观/换肤/档位编辑**：daisyUI 主题切换 + 元素级取色 + 在 Demo 上直接圈改批注。🟢
- **设计建议采纳→生成**：导航/布局/字段/流程类设计建议采纳后**真注入生成链**（非纸面）。🟢

### ③ 真后端生成（两条路） 🟡→🟢

**解决的问题**：Demo 是假的（mock 数据、无权限/无级联/无角色隔离），真交付要有真后端。

**路 B：平台自带通用 CRUD 运行时** 🟢
- LLM 产 Prisma 数据模型 → 平台**自动建表 + 通用 CRUD 后端** → 前端调真接口（`/api/app/:projectId/:resource`）。
- **零编译**：运行时通用 CRUD，新增实体不需要构建。已端到端验证（建表→写入→读回真数据）。
- 局限：v1 无 per-app 鉴权、无自定义业务逻辑、复杂关系弱（适合标准台账/CRUD）。

**路 C：若依（RuoYi-Vue-Plus）codegen** 🟡（本周收尾中）
- IR → 若依代码生成 → 完整 CRUD 模块（Controller/Service/Vue + 菜单 SQL），**带完整 RBAC + 数据权限**。
- 已实测：实体 → 12 文件完整可跑 CRUD 源码；关系补全的产物（主子表/树/中间表）已接进 codegen 输入。
- **数据权限**（"普通用户只看自己/管理员看全部"）= 若依开箱即有、Demo 修不出的那块。
- 收尾中：exploded 部署绕模板加载坑、RBAC 运行时配、provision 串完整链（建表→codegen→build→部署）。
- 适配：**政企信创主力**（若依 MIT + 信创适配 + 多租户）。

> 两路同一套 `BackendRuntime` 接口，前端/传感器/部署不感知差异。路 B 轻快、路 C 重全。

### ④ 自治建造回路 🟢

**解决的问题**：生成中等复杂度程序，单次 LLM 调用不可靠（会截断/负优化）。

- 把 coding-agent 的 SOP（先读后改/有界增量/每步验证/状态外置/断点续跑）编码成**确定性建造编排器**。🟢
- **逐模块测试门**：每个模块生成后过结构门（内容达标 + 是真功能界面而非介绍页），不过不算完成。🟢
- **状态外置 + 续跑**：建造日志落库，跨上下文/被打断后能对账续跑。🟢
- **有界并行**：原子认领 + DAG 依赖，多模块并发建造。🟢
- 核心理念：**严谨在回路、不在单次调用；控制论只把"门"自动化，门真才站得住**。

### ⑤ 交付与上线 🟡

**解决的问题**：交付不只是给个链接，要可运行/可部署/可拿走源码。

- 在线访问链接 + 部署编排（部署时自动 provision 后端 schema）。🟢
- 规划中的交付物：源码下载、项目包导出、Git 仓库交付、数据库结构导出、部署配置导出。🟡（接口/字段预留，未全兑现）

### ⑥ 持续守护中心 🟢

**解决的问题**：交付不是终点，上线后会坏；做"持续守护"=经常性收入地基。

- **定时体检**：上线项目自动入列，每 6h 巡检，复用验收引擎算**健康分**（通过率×0.7 + 传感器×0.3），healthy/degraded/critical 分级。🟢
- **分级自愈**：按风险四级——unknown→提醒、critical→需确认（高风险绝不自动改）、degraded 小幅→低风险自动修、其余→建议。自动修走**快照回滚点 + 劣化即回滚**兜底。🟢
- **月度健康报告**（含前端页）。🟢
- 守护套餐计费 🔴（变现未做）。

### ⑦ 可信底座（贯穿） 🟡

**解决的问题**：政企最在意"可信"——别把用户内部信息/技术细节/越权操作漏出去。

- **多传感器护城河**：L1/L2/L3 + 交叉验证的质量评分引擎（自迭代质量护栏）。🟢
- **数据不出域**：`AI_MODE=local` 全程域内大模型，私有化自托管前端资源（daisyUI/Tailwind 走域内）。🟢
- **交付控制**：路由决策/越界文件/失败文案脱敏三个观察点留痕（当前"观察+留痕"，"拦不拦"是显式开关待加）。🟡
- **多租户地基**：Organization/Membership + tenant-scope + JWT 带 orgId 已铺。🟡（计费/配额未接）
- **用户界面约束**：内部词（hermes/n8n/agent/工程控制论等）+ 内部错误/堆栈/任务 ID **绝不漏给普通用户**。🟢

## 3. 适配场景画像（什么需求容易接）

**强适配**（平台正中靶心）：
- 台账/CRM/巡检/审批/资产/工单等**表单 + 列表 + 角色权限**类业务系统。
- 政企/央国企、要**信创 + 数据不出域 + 私有化**的项目。
- 需要"非技术人员自己提需求、确认、上线"的**降门槛**场景。
- 交付后要**长期托管/运维守护**（按月续费）的场景。
- 有蓝湖/原型截图、要**照着复刻前端**的项目。

**可适配但要补**（有地基、需要工作量）：
- 复杂关系/复杂业务逻辑（走若依 codegen，非通用 CRUD）。
- 行业专属合规规则（行业规则包 🔴 基本空白，要现做）。
- 计费/订阅/配额（多租户地基有，商业化层 🔴 未做）。

**弱适配 / 不在能力圈**（诚实边界）：
- 高并发 C 端互联网产品、实时音视频/IM、复杂算法/大数据计算类。
- 需要大量自定义前端交互、像素级设计还原（复刻能近似、非逐像素）。
- 文件上传、实时推送、深度第三方集成——**外部适配器框架 🔴 未起**（如语音识别/OCR 等外部能力当前无接口）。
- 强一致事务、金融核心交易类。

---

## 4. 技术栈与形态（适配时对方常问）

- 后端 NestJS + Prisma + PostgreSQL；队列 BullMQ；前端 Next.js + daisyUI/Tailwind。
- 生成的业务系统：路 B = 平台通用 CRUD；路 C = 若依 RuoYi-Vue-Plus（Spring Boot 3 + MySQL/信创库 + Vue3）。
- 大模型：DeepSeek（可换），`AI_MODE=local` 支持全本地。
- 部署：私有化独立实例 / SaaS 多租户单实例（两档并存）。

## 5. 适配自查清单（拿去对每个项目打勾）

| 维度 | 问题 | 平台现状 |
|---|---|---|
| 业务形态 | 是不是表单/列表/审批/台账/权限类？ | 🟢 是→强适配 |
| 用户 | 提需求的是不是非技术人员？ | 🟢 正是为此设计 |
| 数据合规 | 要不要数据不出域/信创/私有化？ | 🟢 支持 |
| 权限 | 要不要"按角色/按人"的数据权限隔离？ | 🟢 若依路；🟡 通用 CRUD 路 |
| 关系复杂度 | 主子表/树/多对多够不够？ | 🟢 够；更复杂关系要评估 |
| 交付物 | 要不要源码/Git/数据库结构导出？ | 🟡 部分预留，需确认范围 |
| 运维 | 要不要上线后持续守护/自愈？ | 🟢 有 |
| 行业合规 | 有没有行业专属规则强约束？ | 🔴 要现做行业规则包 |
| 商业化 | 要不要计费/订阅/配额？ | 🔴 地基有、未接 |
| 外部能力 | 要不要 OCR/语音/IM/支付等外部集成？ | 🔴 适配器框架未起 |
| 形态 | C 端高并发/实时/大数据？ | 🔴 不在能力圈 |

---

> 用法建议：把每个候选项目的需求逐条落到第 5 节表格，**🟢 多 = 直接能接；🟡 多 = 能接但要排期；🔴 命中核心诉求 = 慎接或当定制**。需要我针对某个具体项目需求做逐条适配评估，把需求发我，我对着代码现状给结论。
